# TC-48. Чат-вью отображает N участников и маркирует bubble'ы по роли (#0041)

Закрывает US-41. Покрывает рендер чата для сессии с >2 участниками:
шапка-список аватаров, ролевые цвета bubble'ов, компактная строка
`participant_joined`. Поскольку сейчас на e2e-стенде нельзя
автоматически прогнать живую multi-role сессию (для воспроизведения
нужны `team.invite`/`team.escalate` под полную цепочку без интерактива),
TC ручной — собран из проверок на storybook и инспекции UI на реальном
ране, где появилось `participant_joined`.

## Предусловия

- `nvm use && npm run lint && npm run build && npm run test:unit` —
  всё зелёное. Среди unit-сюит присутствует
  `chat/lib/roles.test.ts` с тестами `resolveRoleFrom`,
  `resolveRoleByName`, `participantToRoleInfo`, `formatJoinTime` —
  все кейсы зелёные.

## Шаги

### A. Storybook — визуальная проверка раскладки

1. Запустить `nvm use && npm run storybook` (порт 6006).
2. Открыть категорию **Features → Chat → Multi-participant**.
3. История **Three Participants With Join Event**:
   - В шапке — три аватара (user — зелёный круг с человечком,
     product — синий с пакетом, architect — фиолетовый с компасом).
   - В ленте две первые пары bubble'ов (user, product) с
     соответствующими ролевыми левыми бордерами.
   - Посередине — компактная строка
     «→ Архитектор присоединился в HH:MM» с малым аватаром архитектора
     и стрелкой. **Не** оформлена как bubble.
   - После строки — bubble архитектора с фиолетовым левым бордером.
4. Переключить тему через Storybook toolbar
   (Dark Modern → Light Modern → High Contrast). Проверить, что:
   - Цвета ролевых бордеров и аватаров видимы и различимы во всех
     трёх темах (синий, фиолетовый, оранжевый, зелёный).
   - Текст «присоединился в …» читаем (не сливается с фоном).
5. История **Full Team Room**: в шапке — четыре аватара (user, product,
   architect, programmer); цвет programmer'а — оранжевый, иконка `Code`.
6. История **Pre Join Header Only**: только два участника в шапке —
   проверка, что компонент корректно рендерит малое количество.

### B. Боевая проверка на ране (если есть свежий ран с

`participant_joined`)

1. Открыть VS Code с workspace, в котором есть ран с зафиксированным
   событием `participant_joined` в `tools.jsonl` (можно подложить в
   `.agents/runs/<runId>/<sessionId>/tools.jsonl` синтетическую строку
   `{"kind":"participant_joined","at":"2026-04-26T10:00:00.000Z","role":"architect"}`
   между двумя обычными chat-записями для проверки рендера).
2. Открыть боковую панель `AI Frontend Agent`, выбрать ран, перейти на
   вкладку **Чат**.
3. Шапка чата сразу под прогресс-баром стоимости показывает горизонтальный
   ряд аватаров — все участники сессии.
4. В ленте на месте подложенного события появляется компактная строка
   «→ Архитектор присоединился в HH:MM» (HH:MM — локальное время по
   `at` с пользовательским TZ).
5. Переключиться на другую сессию рана через канвас (drill-in) —
   шапка обновляется на состав именно той сессии. При возврате —
   восстанавливается исходный список.
6. (Опционально, если есть инструменты симуляции) дописать в
   `tools.jsonl` ещё одно `participant_joined` с другой ролью; через
   ~1с (live-watch storage) шапка чата получает дополнительный
   аватар, в ленте появляется новая строка `participant_joined` —
   без перезагрузки webview.

## Ожидание

- Lint, build, test:unit зелёные.
- Storybook-сторис **Three Participants With Join Event** и
  **Full Team Room** рисуются корректно во всех трёх темах VS Code,
  без layout-разрывов, цвета ролей различимы.
- В реальном чате `RunDetails` появляется шапка участников (если
  `participants` непуст) и компактные строки `participant_joined`
  внутри ленты.
- Существующий user-маркер (синий focusBorder для `user`-bubble'ов)
  сохранён.

## Известные ограничения

- Полностью автоматический e2e-сценарий с прогоном модели через
  `team.invite`/`team.escalate` пока невозможен (см. известные
  ограничения TC-44/TC-45/TC-47): перехват OpenRouter в
  `tests/e2e/test-extension/` рассчитан на одиночные ответы,
  multi-role цепочка вручную не собирается. Появится отдельным TC,
  когда e2e-стенд научится прогонять команду из >2 ролей без
  интерактива.
- Live-обновление шапки через настоящий `runs.updated` проверяется
  на ране в шаге B6: автоматического теста нет, но контракт
  обновления подтверждён существующими TC-37 (canvas live-state) и
  работой store/ipc.
