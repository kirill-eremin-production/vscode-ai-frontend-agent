# TC-53. Журнал встреч — карточки сессий рана в боковой панели

Покрывает US-45 (#0046). Ручной кейс: правая side-area получает
второй таб «Встречи», который рендерит хронологический список
карточек сессий рана. Клик по карточке проваливает в чат
выбранной сессии. Автоматизация позже — пока живой ран с
несколькими сессиями требует ручного воспроизведения сценария
TC-44/45 (handoff product → architect → programmer).

## Предусловия

- VS Code открыт в чистом workspace, OpenRouter-ключ задан.
- Расширение собрано (`npm run build`).
- В команду входят минимум 2 роли (`product`, `architect`), чтобы
  при handoff'е появилась bridge-сессия, а не одна корневая.

## Шаги

1. Создать ран, дождаться завершения первой реплики продакта.
   Открыть правую side-area (если свёрнута — кликнуть
   ChevronLeft). Header показывает tab-strip из двух кнопок:
   `Сессии` (активна по умолчанию) и `Встречи`.
2. Нажать кнопку «Встречи» (`button[data-side-panel-tab-button="meetings"]`).
   Активный таб меняется (`aria-selected="true"`), у `<aside>`
   появляется `data-side-panel-tab="meetings"`. Внутри панели
   виден заголовок «Журнал встреч рана» (через `aria-label`).
3. В пустой ран (без сессий) тело панели показывает fallback
   «У этого рана пока нет встреч.». В обычном ране сразу видна
   карточка корневой user↔product сессии:
   - `data-meeting-card` присутствует, `data-session-id` —
     id текущей сессии;
   - `data-meeting-status="active"` (ран в `running` или
     `awaiting_user_input` и сессия — активная);
   - `data-meeting-input-from="user"` отрисован как «← Вы»;
   - время запуска относительное (`Nm ago`, либо `HH:MM` если
     прошло больше часа);
   - превью одной строки = последнее сообщение текущего чата
     (`data-meeting-preview`).
4. Дождаться handoff'а: продакт сдал бриф, появилась
   bridge-сессия product↔architect. В журнале появляется вторая
   карточка СВЕРХУ (свежая по `createdAt`):
   - `data-meeting-input-from="product"` («← Продакт»);
   - её участники — два аватара ролей `product` и `architect`
     (`data-meeting-participant="product"` и `="architect"`);
   - предыдущая корневая карточка опускается ниже — порядок
     сортировки `createdAt desc`.
5. Кликнуть по нижней (корневой) карточке. Главная область
   переключается на чат-таб (`button[data-run-tab="chat"]`
   получает `aria-selected="true"`), `SessionsPanel`/левая колонка
   подсвечивает выбранную сессию — drill-in работает по тому же
   маршруту, что и канвас (#0044/#0045).
6. Свернуть боковую панель ChevronRight. Ширина схлопывается в
   узкий rail, иконка `CalendarDays` остаётся, бейдж сверху
   показывает количество встреч. Развернуть назад — таб
   `Встречи` сохранён (per-run UI prefs, persist `state.setUiPref`).
7. Перезагрузить webview (Reload Webviews). Открыв тот же ран,
   таб `Встречи` всё ещё активен — `sidePanelTabByRun` восстановлен
   из `uiPrefs`.

## Ожидаемый результат

- Все шаги 2–7 выполняются без отклонений.
- Сортировка карточек — свежие сверху; активная — с зелёной
  точкой; завершённые — без неё.
- Клик по любой карточке открывает её сессию в чат-табе.

## Известные ограничения

- Превью сообщения отображается только для ПРОСМАТРИВАЕМОЙ
  (активной) сессии — `selectedDetails.chat` живёт в store
  именно для неё. Для остальных карточек в превью-слот
  подставляется fallback «Встреча N». Per-session preview
  потребует серверного поля `lastMessagePreview` на
  `SessionSummary` — отдельная задача, выходящая за #0046.
- Paused-визуал (`data-meeting-status="paused"`) сейчас
  показывается на сессиях, чей ран в `awaiting_human` —
  это заглушка. Реальный paused-статус сессии и собственная
  иконка появятся в #0052.
- Поиск/фильтр по списку встреч — отдельный tracker (#0047
  prev/next-навигация и search).
