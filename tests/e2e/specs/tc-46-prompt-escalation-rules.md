# TC-46. Промпты ролей: раздел про invite vs escalate (#0039)

В системных промптах ролей (`product`, `architect`, `programmer`) после
#0039 должен быть раздел `## Команда и эскалация`, объясняющий модели
правило выбора между тулами `team.invite` (#0037) и `team.escalate`
(#0038). Сами тулы уже работают (см. TC-44, TC-45) — этот TC проверяет
soft-rule, без которого модель собирает цепочку посредников вручную или
дёргает продакта в обход архитектора.

Запуск тулов делает модель внутри agent-loop'а, и AC задачи #0039 чисто
текстовый: «в файле промпта есть такой-то раздел». Поэтому TC ручной и
проверяет инвариант через исходники + unit-тесты.

## Подготовка

1. `nvm use && npm run lint && npm run build && npm run test:unit`.
   В выводе unit-тестов присутствует сюита
   `prompts.team_escalation` с подсуитами:
   - `общий раздел во всех ролях` — четыре проверки `it.each` для
     `product`, `architect`, `programmer` (заголовок раздела, иерархия,
     упоминание обоих тулов, правило выбора);
   - `programmer-конкретика` — пример `team.escalate('product', ...)`
     и итоговый состав `[programmer, architect, product]`;
   - `architect-конкретика` — `team.invite('product' | 'programmer', ...)`
     и формулировка «escalate ты практически не используешь»;
   - `product-конкретика` — `team.escalate('programmer', ...)`.

## Шаги

1. Открыть `src/extension/entities/run/roles/product.prompt.ts`. Найти
   раздел `## Команда и эскалация`. Проверить:
   - описание иерархии `User → product → architect → programmer`;
   - оба тула упомянуты с подсказкой, на что они похожи (соседи →
     invite, через уровень → escalate);
   - конкретика для роли: для прямого ответа программиста используется
     `team.escalate('programmer', ...)`, архитектор окажется в комнате
     автоматически.
2. Открыть `src/extension/entities/run/roles/architect.prompt.ts`. В
   разделе `## Команда и эскалация` проверить:
   - что соседи — `product` и `programmer`, оба доступны через
     `team.invite('product' | 'programmer', ...)`;
   - явно сказано, что `team.escalate` архитектору практически не нужен
     (в иерархии нет роли «через уровень» от него).
3. Открыть `src/extension/entities/run/roles/programmer.prompt.ts`. В
   разделе `## Команда и эскалация` проверить:
   - конкретный пример вопроса продакту: вызов
     `team.escalate('product', '<вопрос>')`;
   - явно зафиксированный итоговый состав комнаты
     `[programmer, architect, product]` — иллюстрация того, что
     архитектор подтянется автоматически.
4. Сравнить разделы между ролями: формулировка иерархии и базовое
   правило выбора invite/escalate должны быть согласованы (короткий
   повтор в каждом промпте допустим по AC #0039).

## Ожидание

- Lint и build зелёные (изменения только в текстах промптов, типы не
  трогаются).
- Сюита `prompts.team_escalation` зелёная — все четыре подсуиты.
- Записи US-39 в `tests/user-stories.md` на месте.
- В разделах нет упоминаний несуществующих тулов / ролей (например,
  «team.delegate», «manager») — только `team.invite`, `team.escalate`
  и роли из `HIERARCHY` (#0033).

## Известные ограничения

- Полноценный E2E «модель сама выбирает invite/escalate в нужный момент»
  не делается: это поведенческое свойство модели, а правило «smoke на
  реальных моделях не делаем» (AGENT.md) запрещает такую проверку.
  TC-44 и TC-45 уже проверяют сами тулы; этот TC закрепляет наличие
  правила, по которому модель должна между ними выбирать.
- Snapshot-тестов на полный текст промпта намеренно нет — он переживает
  частые точечные правки соседних разделов; ключевые подстроки в
  unit-тестах устойчивее на таких диффах.
