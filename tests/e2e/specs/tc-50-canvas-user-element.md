# TC-50. Canvas — отдельный User-элемент над иерархией с drill-in

Покрывает US-43 (#0043). Ручной smoke на тему «User — заказчик, не
агент»: проверяем, что над иерархией кубиков нарисован круглый
аватар пользователя, к нему идёт тонкая статичная линия от продакта,
и клик по аватару открывает корневую user↔product сессию рана.

## Предусловия

- VS Code открыт в чистом workspace, OpenRouter-ключ задан.
- Расширение собрано (`npm run build`).
- На канвасе после #0042 уже работает иерархическая раскладка
  кубиков (TC-49 проходит).

## Шаги

1. Создать новый ран с произвольным промптом, дождаться, пока
   стартует продакт и появится bridge с архитектором (можно
   использовать тот же сценарий, что в TC-32 multi-agent-hybrid).
2. Перейти на вкладку «Карта команды» (canvas).
3. Визуально проверить раскладку:
   - **Над** верхним кубиком (продакт) виден **круглый** аватар
     пользователя с иконкой человека внутри.
   - Аватар явно меньше кубика по высоте (диаметр < высоты
     кубика) и круглый, не прямоугольный.
   - Аватар выровнен с кубиками **по центру по горизонтали**
     (одна вертикальная ось с продактом).
   - Между аватаром и продактом — **одна тонкая статичная линия**
     того же стиля, что и линии между уровнями (без стрелок, без
     подписей, без пунктира).
4. В DevTools webview'а (Help → Toggle Developer Tools) открыть
   Inspector и убедиться:
   - У User-элемента есть атрибут `data-canvas-user`.
   - У User-элемента есть атрибут `data-canvas-drill-session` со
     значением — id корневой сессии рана (та, у которой
     `inputFrom === 'user'` и пустой `prev`). Это тот же id, что
     открывается по клику на кубик продакта (drill-in продакта
     ведёт в ту же корневую сессию).
   - Линия от User имеет атрибут
     `data-canvas-reporting-line="user--product"`.
5. Активировать User-элемент (фокус → Enter, либо клик):
   - Открывается вкладка «Чат» с корневой user↔product сессией
     (в `SessionsPanel` подсвечивается строка с этим
     `data-session-id`).
   - В ленте видно исходное сообщение пользователя
     (`from: "user"`), которое было задано при создании рана.
6. Вернуться на «Карту». Активировать кубик продакта — должна
   открыться **та же самая** сессия, что и при клике по User
   (drill-in продакта по контракту owner-based ведёт в его
   корневую сессию).
7. Сценарий «корневой сессии нет» проверяется юнит-тестами на
   `resolveUserDrillSession` (см. `drill-resolver.test.ts`):
   ручной воспроизводимый кейс отсутствует, потому что в живом
   ране корневая сессия создаётся всегда.

## Ожидаемый результат

Все шаги 3–6 выполняются без отклонений. User-элемент явно
отделён визуально от кубиков агентов, drill-in работает и ведёт в
корневую сессию.

## Известные ограничения

- Аватар — статичная иконка (lucide-react `User`); аватар
  профиля пользователя пока не загружается.
- Состояние «User что-то ввёл/печатает» на этом элементе не
  отображается (live-индикатор у User-элемента не предусмотрен).
- Hybrid-режим (#0012, US-30) добавляет user-участника в
  bridge'и; кубик/маркер для этого случая — отдельная задача,
  не покрывается этим TC.
