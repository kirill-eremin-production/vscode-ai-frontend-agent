# TC-54. Журнал встреч — prev/next-ссылки и навигация между карточками

Покрывает US-46 (#0047). Ручной кейс: каждая карточка встречи в
журнале (#0046) получает компактные prev/next-ссылки на соседние
сессии. Клик по ссылке скроллит панель к нужной карточке и
подсвечивает её визуально на ~1.5s, без открытия чата. Orphan-ссылка
(целевая сессия не найдена) рендерится disabled с tooltip'ом «сессия
не найдена».

## Предусловия

- VS Code открыт в чистом workspace, OpenRouter-ключ задан.
- Расширение собрано (`npm run build`).
- В команду входят минимум 3 роли (`product`, `architect`,
  `programmer`), чтобы получить цепочку из ≥3 сессий с непустыми
  prev/next (root user↔product, bridge product↔architect,
  bridge architect↔programmer). Сценарий повторяет TC-44/45 handoff'ов.

## Шаги

1. Создать ран, дождаться завершения первой реплики продакта. Открыть
   правую side-area, переключить таб на «Встречи»
   (`button[data-side-panel-tab-button="meetings"]`).
   Видна одна карточка корневой user↔product сессии:
   - `data-meeting-card` присутствует, `data-session-id` — id сессии;
   - prev/next-строк ещё нет (root: `prev=[]`, `next=[]`).
2. Дождаться handoff'а: продакт сдаёт бриф, появляется bridge-сессия
   product↔architect. В журнале две карточки. У свежей (сверху):
   - есть строка `← откуда:` (`data-meeting-link-row="prev"`);
   - внутри — кнопка `data-meeting-link="prev"` с
     `data-meeting-link-target=<id корневой сессии>`;
   - tooltip кнопки начинается с `HH:MM` корневой сессии.
     У корневой (снизу) появляется строка `→ что родилось:`
     (`data-meeting-link-row="next"`) с симметричной ссылкой.
3. Дождаться второго handoff'а — bridge architect↔programmer. Теперь
   три карточки; у средней (architect-bridge) есть и prev, и next.
4. Кликнуть по next-ссылке корневой карточки: ссылка ведёт на
   bridge-карточку. Панель должна:
   - проскроллить к bridge-карточке так, чтобы она оказалась в центре
     видимой области (`scrollIntoView({block:'center', behavior:'smooth'})`);
   - подсветить её визуально (`data-meeting-flash="on"` появляется
     на карточке);
   - чат-таб **не** открыться — фокус остаётся в журнале
     (`button[data-run-tab="chat"]` сохраняет прежнее `aria-selected`).
5. Через ~1.5s атрибут `data-meeting-flash` исчезает; outline
   карточки возвращается к прозрачному. Если кликнуть по другой
   ссылке быстрее, чем сработает таймер, подсветка переключается на
   новую карточку без мерцания (старый таймер сбрасывается).
6. Кликнуть по prev-ссылке у самой свежей (programmer-bridge)
   карточки — фокус скроллится к предыдущей (architect-bridge).
   Строка участников ссылки рендерит `data-meeting-link-icon` для
   каждой роли участника целевой сессии без подписей-имён.
7. Hover'ом задержаться на любой prev/next-ссылке: tooltip
   показывает `HH:MM · <первое сообщение>` — но только для тех
   ссылок, чья целевая сессия совпадает с текущей просматриваемой
   (chat хранится в store только для неё). У остальных tooltip
   ограничивается одним временем. Это известное ограничение
   (см. Outcome #0046, симметрично с per-session preview).
8. Сценарий orphan: вручную открыть `.agents/runs/<id>/meta.json`,
   у одной из сессий заменить элемент `next[]` на несуществующий
   id (например, `"orphan-x"`), сохранить файл и перевыбрать ран в
   списке (или Reload Webviews). У соответствующей карточки в
   `→ что родилось:` появляется ссылка
   `data-meeting-link="next"` c `data-meeting-link-disabled="true"`,
   `aria-disabled="true"`, tooltip «сессия не найдена». Клик по ней
   ничего не делает (нет навигации, нет flash).
9. Клик по самой карточке (не по prev/next-кнопке внутри неё)
   по-прежнему открывает чат-таб этой сессии (drill-in работает,
   как в US-45, регрессии нет).

## Ожидаемый результат

- Все шаги 2–9 выполняются без отклонений.
- prev/next-навигация работает только внутри панели «Встречи» — чат
  не открывается, side-area-таб не переключается.
- Подсветка карточки длится ровно ~1.5s и снимается без мерцания
  при повторных кликах.
- Orphan-ссылка визуально отличается (border-dashed, opacity), не
  кликабельна, имеет понятный tooltip.

## Известные ограничения

- Per-session «первое сообщение» доступно только для просматриваемой
  сессии — для остальных tooltip ограничивается временем (см.
  Outcome #0046, лечится серверным полем `firstMessagePreview`).
- В сторонних темах VS Code outline-цвет flash'а берётся из
  `--vscode-focusBorder`; если тема его не определяет, fallback —
  фиксированный `#0078d4` (см. MeetingCard.tsx).
- Поиск/фильтр по списку встреч за пределами #0047 — будут отдельной
  задачей.
