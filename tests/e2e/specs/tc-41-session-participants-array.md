# TC-41. participants — массив произвольной длины (#0034)

Задача чисто типовая (миграция формата `Session.participants` с пары
на массив длины ≥ 1) и прямого UI-эффекта не имеет. Этот TC ручной и
сводится к двум проверкам: что storage корректно нормализует старые
session-meta при чтении, и что новые сессии пишутся в новом формате.

## Подготовка

1. `nvm use && npm run lint && npm run build && npm run test:unit`.
   В выводе unit-тестов присутствует сюита
   `participants — массив произвольной длины (#0034)`, все её кейсы
   зелёные (нормализация legacy meta + round-trip длины 2 и 3).

## Шаги

1. Запустить расширение из VS Code (F5) на тестовом workspace.
2. Создать новый ран (US-2) — продакт стартует. Завершить его до конца
   (получить `brief.md`, ран `done`). Это создаст новый session-meta
   в новом формате.
3. Открыть `.agents/runs/<runId>/sessions/<sid>/meta.json` в редакторе.
4. В исходниках сделать
   `grep -rn "participants\\[\\|participants\\.length === 2" src/`.

## Ожидание

- Шаг 1: lint, build, unit зелёные. В юнит-логах есть кейсы
  `readSessionMeta нормализует legacy meta.json без participants`,
  `создание сессии с парой участников`, `создание сессии с тройкой
участников (комната)`.
- Шаг 3: в `meta.json` нового рана есть поле `participants` с массивом
  объектов `{ kind: 'user' }` / `{ kind: 'agent', role: 'product' }`.
  Поля `agentRole` нет.
- Шаг 4: ни одного попадания на `participants[0]/[1]` или
  `participants.length === 2` в `src/`. Допустимы только проверки
  на пустоту (`length === 0`) и обходы через `some/filter/map`.

## Известные ограничения

- Авто-кейс на нормализацию старых файлов не делаем: e2e ставит чистый
  workspace, легаси-файлы там не появятся естественно. Поведение
  закрыто unit-тестом, который пишет старый формат на диск напрямую.
