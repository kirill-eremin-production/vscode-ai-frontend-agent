# TC-47. Resumer — слайсинг истории по participant_joined для роли (#0040)

Задача инфраструктурная: резумер при подъёме сессии-комнаты подаёт
каждой роли её «личный срез» истории — события до её
`participant_joined` сворачиваются в system-блок «контекст до твоего
прихода» с system-маркером, а более поздние идут обычной лентой. UI
чата задачей не меняется (рендер событий — в #0041), e2e-сценария на
рантайме модели сейчас построить нельзя (для воспроизведения нужны
тулы `team.invite`/`team.escalate` под полный multi-role прогон без
интерактива). Поэтому TC ручной — проверяет поведение через unit-сюиту
и инспекцию файлов.

## Подготовка

1. `nvm use && npm run lint && npm run build && npm run test:unit`.
   В выводе unit-тестов присутствует сюита
   `reconstructHistory — slicing по participant_joined (#0040)`, все её
   кейсы зелёные:
   - `для роли с participant_joined: события до joinedAt уходят в
pre-блок, далее маркер, далее post-секция`
   - `для роли-«старожила» без participant_joined: вся история одной
лентой + событие входа другой роли видно как system`
   - `идемпотентность: повторный вызов с теми же входами даёт
идентичный результат`
   - `pre-блок включает tool_calls и tool_results в текстовом виде`
   - `participant_joined с at === joinedAt попадает в post-секцию`
   - `пустой pre-блок не добавляется в историю (когда до joinedAt
релевантных событий нет)`

## Шаги

Поскольку без живого мульти-ролевого прогона полный кейс с моделью
сейчас не собрать, TC сводится к verification по результатам
unit-сюиты + инспекции исходников.

1. Открыть `src/extension/shared/agent-loop/resume.ts`.
2. Убедиться, что в `reconstructHistory` есть вызов
   `findParticipantJoinedAt(events, config.role)` и развилка:
   - если `joinedAt !== undefined` → строится pre-блок (через
     `renderPreJoinBlock`), затем `PARTICIPANT_JOIN_MARKER`, затем
     post-события через `appendEventAsMessage`;
   - иначе все события идут через `appendEventAsMessage` как раньше.
3. Открыть `tests/user-stories.md` и убедиться, что US-40 описывает
   контракт.
4. (Опционально, если есть свежий ран с `participant_joined` в
   `tools.jsonl`.) Симулировать рестарт VS Code: закрыть и снова открыть
   workspace, выбрать ран в сайдбаре. В чате должна остаться вся лента
   (UI слайсинг не делает — это поведение #0040 не затрагивает); в
   логах extension host'а через лог рана можно увидеть, что resumer
   стартует, но визуально ничего нового не появится.

## Ожидание

- Lint и build зелёные — TS-границы webview/extension не нарушены.
- Unit-сюита `reconstructHistory — slicing по participant_joined
(#0040)` зелёная.
- В `tests/user-stories.md` есть запись US-40 с описанием
  инфраструктурного контракта.
- В чате `RunDetails` ничего видимо не меняется (по требованию issue:
  «Без изменений в UI чата (рендер событий — в #0041)»).

## Известные ограничения

- Полноценный e2e (создать ран, запустить продукта, через
  `team.escalate` подтащить программиста, перезапустить VS Code,
  убедиться, что программист в LLM-контексте получил pre-блок) можно
  собрать только когда e2e-стенд научится прогонять цепочку ролей без
  интерактива. Сейчас перехват OpenRouter из
  `tests/e2e/test-extension/` рассчитан на одиночные ответы, не на
  мульти-ролевые сессии. Эту цепочку добавим отдельным TC, как только
  появится сценарий запроса/ответа с участием нескольких ролей.
- UI-рендер `participant_joined` в чате — задача #0041, проверяется
  своим TC.
