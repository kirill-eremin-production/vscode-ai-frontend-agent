# User stories

Истории описывают, что пользователь хочет получить от расширения на текущем этапе. К каркасу рана (создать, увидеть в списке, посмотреть детали) добавилась первая полноценная роль — **продакт** (#0003): после старта рана он читает свою kb, при необходимости задаёт уточняющие вопросы и выдаёт финальный `brief.md`. Архитектор и программист — следующими задачами.

US-10–US-16 закрепляют фундаментальные требования к UX чата и к личности продакта: продолжение диалога, прозрачность тулов, стоимость/контекст, вопросно-ориентированный режим работы, запрет на технические решения, универсальность и документирование истории изменений.

**Концептуальный сдвиг 2026-04-26 (US-28..US-33):** изначальная идея показывать рабочий процесс и все коммуникации как flow со стрелочками признана нерабочей. Причина: участники общаются друг с другом много раз, стрелки переплетаются, и хуже того — flow-картинка теряет ключевую информацию «кто когда и с кем взаимодействовал». На замену flow приходят два комплементарных вида: **canvas как иерархия команды** (org-chart, кубики со статусами, не растёт с числом коммуникаций) и **хронологический журнал встреч** (лента чатов с участниками, prev/next-связями). Метафора — реальная команда людей: org-chart на стене + журнал встреч в календаре. US-24..US-27 описывают предыдущую flow-модель и **частично замещаются** US-28..US-33 — оставлены как контекст «почему мы пришли к нынешней модели».

## US-1. Сохранить ключ OpenRouter

**Как** пользователь, впервые открывший расширение,
**я хочу** один раз ввести ключ OpenRouter,
**чтобы** дальше расширение могло обращаться к моделям без повторных вопросов.

Acceptance:

- Ключ сохраняется в VS Code SecretStorage (а не в settings.json и не в `.agents/`).
- После закрытия и повторного открытия VS Code расширение помнит ключ.
- Ключ можно перезаписать в любой момент через команду или кнопку.

## US-2. Создать новый ран по запросу

**Как** пользователь с открытым проектом,
**я хочу** ввести произвольный текст задачи и нажать «Start run»,
**чтобы** расширение завело новый ран, присвоило ему короткий осмысленный заголовок и сразу же отдало работу продакту.

Acceptance:

- Заголовок генерируется дешёвой моделью через OpenRouter (3–6 слов, на языке запроса).
- Если генерация заголовка упала — ран всё равно создаётся, заголовок берётся из первых символов запроса.
- На диске появляется `.agents/runs/<runId>/meta.json` и `chat.jsonl` с первым сообщением `from: "user"`.
- Сразу после создания ран переходит в `running` и стартует продакт (US-7); пользователь не должен дополнительно нажимать кнопку «запустить агента».

## US-3. Видеть список своих ранов

**Как** пользователь,
**я хочу** видеть слева список всех ранов в текущем проекте,
**чтобы** возвращаться к своим задачам после перезапуска VS Code.

Acceptance:

- Свежие раны сверху.
- В каждой строке — заголовок, дата создания и статус.
- Список выживает перезапуск VS Code (читается с диска).

## US-4. Открыть детали выбранного рана

**Как** пользователь,
**я хочу** кликнуть по ранy в списке,
**чтобы** увидеть его исходный запрос и ленту сообщений.

Acceptance:

- Карточка справа показывает заголовок, статус, исходный prompt и список сообщений из `chat.jsonl`.
- Если на диске есть `brief.md` (артефакт продакта, US-7) и/или `plan.md` (артефакт архитектора, #0004) — оба показываются последовательными секциями над лентой сообщений.
- Если выбранную папку рана удалить руками — UI показывает «Загружаю…», а не падает.

## US-5. Безопасное поведение без workspace и без ключа

**Как** пользователь, открывший VS Code без папки или без ключа,
**я хочу** получить понятное сообщение,
**чтобы** не гадать, почему ничего не происходит.

Acceptance:

- Без открытой папки попытка создать ран → ошибка с подсказкой «Откройте папку…».
- Без ключа попытка создать ран → input box; если отменить — ран не создаётся, ошибки нет.

## US-6. Открыть расширение в полноразмерной вкладке

**Как** пользователь, которому тесно в боковой панели,
**я хочу** открыть тот же UI как обычную вкладку редактора,
**чтобы** работать с большим списком ранов и длинным prompt.

Acceptance:

- Кнопка «Open in Tab» / команда `AI Frontend Agent: Open in Tab` открывает панель.
- В панели та же функциональность, что в сайдбаре (создание, список, детали).
- Состояние не теряется при переключении вкладок (`retainContextWhenHidden`).

## US-7. Получить продуктовый бриф из своего запроса

**Как** пользователь, описавший задачу в свободной форме,
**я хочу** через несколько секунд увидеть структурированный бриф, который понимает дальнейший пайплайн (архитектор, программист),
**чтобы** не оформлять требования вручную и не повторять одну и ту же контекстную работу из рана в ран.

Acceptance:

- После старта рана статус идёт `draft → running → … → awaiting_human` (или `awaiting_user_input`, если продакт спросил).
- В `.agents/runs/<runId>/brief.md` лежит финальный markdown с заголовком и фиксированными секциями: `## Проблема`, `## Целевой пользователь и сценарий`, `## User stories`, `## Acceptance criteria`, `## Не-цели`, `## Связанные артефакты kb`. Секции `## Открытые вопросы` нет.
- В карточке рана справа появляется блок «Бриф» с этим текстом, поверх ленты чата.
- В `chat.jsonl` появляется сообщение `from: "agent:product"` с превью брифа (первые 600 символов) — для человекочитаемой ленты.
- Если модель закончила без текста (assistant.content пустой) — ран `failed`, в `chat.jsonl` диагностика «Продакт закончил без текста брифа…», файла `brief.md` нет.

## US-8. Ответить на уточняющий вопрос продакта

**Как** пользователь, описавший задачу неполно,
**я хочу** что продакт сам спрашивал недостающие детали в карточке рана,
**чтобы** не получать в ответ выдуманные требования и не вспоминать заранее всё, что нужно указать.

Acceptance:

- Когда продакт зовёт `ask_user`, ран переходит в `awaiting_user_input`, в карточке появляется форма с вопросом.
- Один вопрос = один фокус (продакт не задаёт «составной» вопрос про всё сразу — это закреплено в его system prompt).
- После ввода ответа и нажатия «Ответить» цикл продолжается, статус возвращается в `running`.
- Если пользователь закрыл VS Code, не отвечая, и потом открыл снова: ран остаётся в `awaiting_user_input`, форма с вопросом видна, ответ возобновляет цикл (resume через `tools.jsonl` + `loop.json`).
- Каждый цикл вопрос-ответ присутствует в `chat.jsonl` парой сообщений `agent:product` (вопрос) и `user` (ответ) — лента читаема глазами.

## US-9. Использовать продуктовую память между ранами

**Как** пользователь, ведущий продукт неделями,
**я хочу** чтобы продакт помнил продуктовые решения, термины и описания фич из прошлых ранов,
**чтобы** новый бриф по уже знакомой области не противоречил старым решениям и опирался на них.

Acceptance:

- Продакт читает `.agents/knowledge/product/` (через `kb.list`/`kb.grep`/`kb.read`) в начале каждого рана; релевантные пути попадают в `## Связанные артефакты kb` финального брифа.
- Продакт пишет новые артефакты только в свою kb (`.agents/knowledge/product/<schema-subdir>/...`): `decisions/YYYY-MM-DD-<slug>.md`, `features/<slug>.md`, `questions/<slug>.md` и т.п.
- Попытка модели записать в чужую kb или в корень `.agents/knowledge/` физически невозможна — пути префиксуются ролью на уровне обёртки тулов (sandbox), это не доверие к prompt'у.
- Файлы продакта коммитятся в git как часть проекта (нет `.gitignore` на эту папку), поэтому история продуктовых решений живёт вместе с кодом.

## US-10. Продолжать диалог с агентом после ответа

**Как** пользователь, получивший от агента готовый артефакт (например, `brief.md` от продакта),
**я хочу** в карточке рана иметь свободное поле ввода для следующего сообщения,
**чтобы** уточнять, поправлять или развивать результат, не создавая новый ран.

Acceptance:

- В карточке рана со статусом `awaiting_human` (артефакт готов) есть свободное поле ввода и кнопка «Отправить» — это не только форма ответа на конкретный `ask_user`-вопрос, а полноценный продолжающийся чат.
- После отправки ран снова переходит в `running`, агент той же роли получает сообщение как очередной `user`-message и продолжает свой цикл (kb-чтение → возможные `ask_user` → обновление kb → новая версия артефакта).
- Сообщение пользователя и последующая активность дописываются в тот же `chat.jsonl` — история диалога не разрезается на «первый бриф» и «доработка», лента читается одной хронологией.
- Артефакт (`brief.md` и т.п.) после нового цикла перезаписывается атомарно — на диске всегда последняя версия; прежние редакции остаются только в git.
- Если ран в `awaiting_user_input` (агент задал конкретный вопрос), то же поле ввода используется и для ответа на этот вопрос (см. US-8) — отдельной формы не плодим.
- После `failed` отправка нового сообщения тоже допустима: ран возобновляется с накопленным контекстом, пользователь не теряет историю из-за временного сбоя.

## US-11. Видеть вызовы инструментов и переходить к созданным файлам

**Как** пользователь,
**я хочу** видеть в карточке рана какие инструменты и с какими аргументами вызывает агент, и одним кликом открывать файлы, которые он создал или изменил,
**чтобы** контролировать ход работы и сразу проверять результат, а не искать артефакты в файловой системе руками.

Acceptance:

- Каждый tool call отображается в общей ленте чата отдельной записью: имя тула, краткие аргументы (полный JSON раскрывается по клику), краткий результат или явная пометка «ошибка». Записи идут в естественной хронологии между сообщениями `agent:*` и `user`, а не в отдельном лог-табе.
- Источник правды — `tools.jsonl`; webview подписан на новые записи и перерисовывает ленту без перезапуска панели.
- Для тулов, создающих или меняющих файлы в workspace (`kb.write` сейчас, в будущем — запись `brief.md`, `plan.md`, файлов кода), запись содержит кликабельный относительный путь. Клик открывает файл новым редактором VS Code (через `vscode.open`/`workspace.openTextDocument`), как обычный файл проекта.
- Если тул упал — запись подсвечена ошибкой и показывает сообщение из `tools.jsonl`, чтобы понять причину без чтения файла руками.
- Конфиденциальные части аргументов (например, содержимое `kb.write.body` целиком) по умолчанию свёрнуты до превью — лента не превращается в стену текста, но «развернуть» доступно одним кликом.

## US-12. Видеть стоимость запросов и заполненность контекста

**Как** пользователь,
**я хочу** видеть стоимость каждого обращения к модели и текущую заполненность её контекстного окна,
**чтобы** оценивать расходы по ходу работы и вовремя сжимать контекст ради экономии.

Acceptance:

- В карточке рана отображается суммарная стоимость рана в USD (посчитанная из `usage` ответа OpenRouter и тарифа конкретной модели) с разбивкой по ролям (`product`, в будущем — `architect`, `programmer`, …).
- Каждое assistant-сообщение в ленте подписано стоимостью этого конкретного шага и количеством input/output токенов; tool-вызовы агрегируются в стоимость соответствующего шага.
- Рядом — индикатор «токенов в контексте / лимит модели» с цветовой шкалой (зелёный → жёлтый → красный по мере приближения к лимиту), чтобы исчерпание контекста не наступало внезапно.
- Когда контекст приближается к лимиту (или по явному желанию пользователя), доступна кнопка «Сжать контекст»: текущая лента сводится моделью в короткое summary, следующий шаг стартует с этого summary вместо полной истории. До сжатия — подтверждение; после — запись в `chat.jsonl` «контекст сжат», прежний `chat.jsonl` сохраняется как `chat.jsonl.<timestamp>.bak` рядом, чтобы ничего не терялось безвозвратно.
- Накопленная стоимость и токены сохраняются на диске (в `meta.json` или соседнем `usage.jsonl`) — переживают перезапуск VS Code и накапливаются за весь жизненный цикл рана, включая повторные циклы из US-10.
- Если для модели в коде не зафиксирован тариф — стоимость показывается как «—» с подсказкой, а не как `0` (чтобы не вводить в заблуждение).

## US-13. Продакт задаёт вопросы, пока пользователь готов отвечать

**Как** пользователь, общающийся с продактом,
**я хочу** чтобы по умолчанию продакт бесконечно задавал уточняющие вопросы и не принимал продуктовых решений за меня,
**чтобы** я мог влиять на каждый значимый выбор и не получать в брифе выдуманные требования. И только когда я явно скажу «хватит вопросов, отдавай в разработку», продакт может закрыть оставшиеся пробелы своими допущениями.

Acceptance:

- Пока пользователь явно не дал сигнал «оформляй бриф», продакт не финализирует `brief.md`: при любом значимом пробеле он зовёт `ask_user` и возвращает ран в `awaiting_user_input`. «Значимый пробел» включает (минимум): нечёткую проблему, не названного целевого пользователя, отсутствие основного сценария, неопределённые acceptance, нерешённые продуктовые альтернативы.
- В карточке рана есть явная кнопка «Достаточно вопросов, оформляй» (доступна и в `awaiting_user_input`, и в `running`). Её нажатие отправляет в чат специальный `user`-message с фиксированным маркером, который продакт распознаёт по system prompt: после него он перестаёт спрашивать и финализирует артефакт самостоятельно.
- Все допущения, которыми продакт закрыл оставшиеся пробелы после такого сигнала, явно фиксируются: либо отдельным разделом в финальном артефакте, либо записью в `decisions/YYYY-MM-DD-<slug>.md` с пометкой «assumption, не подтверждено пользователем».
- Поведение «спрашивай, не решай» закреплено в system prompt продакта (а не в коде) — но кнопка-сигнал и распознавание маркера живут в коде, чтобы пользователь не подбирал «магические слова» в свободном тексте.
- TC проверяет: ран без явного сигнала не уходит в `awaiting_human`, пока в брифе остаются пробелы, отмеченные в acceptance выше; ран с сигналом — уходит, причём допущения зафиксированы.

## US-14. Продакт не принимает технических решений

**Как** пользователь,
**я хочу** чтобы продакт не описывал какие технологии использовать, какие фреймворки/библиотеки выбирать, какую архитектуру строить,
**чтобы** технические решения оставались за архитектором и программистом — продакт отвечает на вопрос «зачем», а не «как».

Acceptance:

- В `brief.md` нет секций про стек, архитектуру, библиотеки, паттерны кода, схемы БД, формат API. Это и так закреплено фиксированным форматом брифа из #0003, но дополнительно явно описано в system prompt продакта как запрет.
- Допустимы только верхнеуровневые продуктовые рамки: тип продукта (веб / мобильное / десктоп / CLI / физический девайс), наличие сети, требование offline и т.п. — без выбора конкретных технологий внутри рамки. Пример из обсуждения: «игра в браузере vs мобильная игра» — допустимо; «React + Redux + IndexedDB» — нет.
- Если пользователь сам спрашивает у продакта про технологии — продакт отвечает в духе «это решит архитектор/программист, я фиксирую только продуктовое ограничение, если оно для тебя важно (например, обязательная работа без сети)», и при необходимости заводит соответствующую запись в kb как продуктовое требование, а не техническое решение.
- В kb продакт фиксирует продуктовые решения; технические упоминания, всплывшие в диалоге, либо переводятся в продуктовую формулировку («важно работать без сети»), либо явно отмечаются как контекст пользователя без принятия решения продактом.
- TC проверяет: на запрос «выбери базу данных / фреймворк / язык» продакт отказывается принимать решение и переводит разговор в продуктовое русло; в `brief.md` нет упоминаний конкретных технологий.

## US-15. Продакт универсален по аспектам продукта

**Как** пользователь,
**я хочу** обсуждать с продактом не только разработку фичи, но и дизайн, маркетинг, целеполагание, монетизацию, бизнес-модель, аналитику,
**чтобы** иметь одного собеседника по всему продукту, а не «другую роль» под каждый аспект.

Acceptance:

- Стартовый prompt пользователя может быть про любой аспект продукта (новая фича, маркетинговая гипотеза, редизайн, pricing, OKR/цели, аналитика поведения и т.п.). Продакт принимает его и работает по одному и тому же пайплайну: kb → ask_user → kb.write → финальный артефакт.
- system prompt описывает продакта именно как «директора продукта» с ответственностью за весь жизненный цикл, а не как «писателя ТЗ для разработки».
- Структура kb продакта (см. #0002) расширяется так, чтобы вмещать все эти классы: помимо `features/`, `decisions/`, `questions/` появляются разделы под маркетинг, дизайн-направление, цели/метрики, ценообразование и т.п. Конкретные имена разделов финализируются при реализации US-16, но схема явно описана в `README.md` kb как открытая.
- Имя и формат финального артефакта зависят от типа рана: `brief.md` для фич, `proposal.md`/`decision.md`/`plan.md` для не-фичевых обсуждений. Список вариантов и правила выбора зафиксированы в system prompt продакта (фиксированный enum, не свободный выбор).
- В карточке рана отображается тип/имя финального артефакта, чтобы пользователь сразу видел, чем закончился ран — а не предполагал, что любой ран обязан произвести `brief.md`.
- TC проверяет: ран с маркетинговым/дизайн-/целевым запросом доходит до соответствующего артефакта без попыток превратить его в техническое ТЗ; релевантные разделы kb создаются и наполняются.

## US-16. Продакт ведёт историю продуктовых решений и помечает устаревшее

**Как** пользователь, ведущий продукт неделями и месяцами,
**я хочу** чтобы продакт документировал каждое значимое решение, связывал его с предыдущими и явно помечал устаревшее новыми ссылками и датой,
**чтобы** в любой момент в kb был полный, согласованный и обоснованный срез актуального состояния продукта, а не россыпь противоречивых заметок.

Acceptance:

- На старте каждого рана продакт обязан (правило в system prompt):
  - прочитать релевантные разделы своей kb (`kb.list` + `kb.grep`/`kb.read`);
  - явно сравнить новый запрос с уже принятыми решениями;
  - если новый запрос конфликтует с прежним решением — создать новый decision в `decisions/YYYY-MM-DD-<slug>.md` с обоснованием смены курса; в каждом «вытесняемом» старом артефакте обновить frontmatter (`status: superseded`, `superseded_by: <путь>`, `superseded_at: YYYY-MM-DD`); в новом decision явно перечислить `supersedes: [<пути>]`.
  - если в затронутой области есть «висящие» противоречия (несколько активных decision-ов друг против друга) — первым шагом задать `ask_user` с просьбой подтвердить актуальность, а не делать вид, что их нет.
- Устаревшие артефакты не удаляются — только помечаются. История решений и причин смены курса остаётся в kb (и в git) и доступна для будущих ранов и для пользователя.
- Каждое решение в kb содержит не только «что решили», но и «почему»: контекст, рассмотренные альтернативы, что отвергли и по какой причине. Заглушки вида «решено: X» без обоснования — нарушение, фиксируется в system prompt как недопустимое.
- Каждая фича/раздел kb имеет в начале файла актуальное описание и явный список ссылок на decisions, которые её сформировали. При смене решения обновляется и описание фичи (а не только заводится новый decision в стороне).
- Финальный артефакт рана в секции `## Связанные артефакты kb` обязан перечислить и те kb-пути, которые продакт прочитал, и те, которые создал/обновил/пометил устаревшими — чтобы из артефакта рана можно было «спрыгнуть» в полную продуктовую историю.
- Пример из обсуждения (закрепить в TC): сначала был decision «делаем игру про средневековье», затем ран «теперь делаем игру про космос» → новый decision со ссылкой `supersedes`, старый decision и связанные feature-файлы перешли в `status: superseded` с датой и ссылкой на новый.
- Это «пытливое» поведение проверяется TC: ран по теме, по которой уже есть зафиксированное решение, должен либо явно сослаться на старое решение (если новое его не отменяет), либо корректно оформить supersede-цепочку (если отменяет).

## US-17. Стильный связный UI с уважением темы VS Code

**Как** пользователь расширения,
**я хочу** чтобы UI выглядел как современное приложение, а не как голый прототип, и при этом естественно вписывался в любую тему VS Code,
**чтобы** длительная работа в нём не утомляла, а ощущение продукта было цельным.

Acceptance:

- Все цвета берутся из CSS-переменных VS Code (`--vscode-foreground`, `--vscode-editor-background`, `--vscode-button-background`, …) через семантический слой токенов (`bg-surface`, `text-muted`, `border-subtle`, `accent`, `danger`, …). Прямых hex/rgb значений в компонентах нет.
- Смена темы VS Code (light → dark → high-contrast) применяется без перезапуска webview и без визуальных регрессий.
- Шрифт, размеры и базовый ритм отступов согласованы по всему UI (одна шкала, например `space-1 = 4px`).
- Иконки берутся из единой библиотеки (lucide-react) — нет смеси эмодзи, ASCII-символов и SVG-инлайнов в проде. Эмодзи допустимы только как осмысленные знаки в данных пользователя или в специальных бейджах (например, «🤝 Передача» как маркер bridge).
- Атомы UI (`Button`, `IconButton`, `Spinner`, `Skeleton`, `Badge`, `Panel`, `Collapsible`, `EmptyState`, `Avatar`, `Tooltip`) живут в одной библиотеке и используются повсеместно — повторных «местечковых» реализаций нет.

## US-18. Список ранов — навигация, а не форма

**Как** пользователь, у которого накопилось много ранов,
**я хочу** видеть список как чистый навигационный сайдбар без формы создания, и иметь возможность скрыть его одним кликом,
**чтобы** освобождать место под основную работу и не путать «начало нового рана» с «выбором существующего».

Acceptance:

- Левая панель — список ранов с заголовком («Раны»), кнопкой «+ Новый ран» в шапке и кнопкой свернуть/развернуть. Поля ввода в этой панели нет.
- Состояние «свёрнута/развёрнута» переживает перезапуск VS Code (хранится в `globalState`/`localStorage`).
- В свёрнутом виде остаётся узкая «полоска» с кнопкой развернуть и иконкой «+ Новый ран» (доступ к созданию рана не теряется).
- Клик по кнопке «+ Новый ран» в шапке списка переводит main-area в режим создания рана (см. US-19), не показывая форму внутри сайдбара.
- В каждой строке списка по-прежнему — заголовок, дата, статус (US-3), плюс компактный индикатор активной роли («работает: продакт»), если ран в `running`.

## US-19. Создание рана — отдельный осознанный шаг в main-area

**Как** пользователь,
**я хочу** что создание нового рана было отдельным экраном в основной области, а не строкой ввода в сайдбаре,
**чтобы** я мог спокойно сформулировать задачу, видеть её целиком и понимать, что это начало новой работы.

Acceptance:

- Когда не выбран ни один ран и нет активного «нового рана» — main-area показывает empty state: иллюстрация/иконка, краткое объяснение, что делает расширение, и primary-кнопка «Новый ран».
- Клик «Новый ран» (из empty state или из шапки списка) показывает форму: textarea для запроса (минимум 8 строк), необязательное поле «Заголовок» (если пусто — генерится моделью, как в US-2), кнопка «Запустить», кнопка «Отмена».
- Форма валидируется: пустой запрос → кнопка disabled с tooltip-объяснением.
- На время `runs.create` IPC-запроса кнопка показывает спиннер с подписью «Создаю ран…»; после успеха — main-area переключается на детали свежесозданного рана, в списке слева он появляется и подсвечен.
- Если ключа OpenRouter нет — UX из US-1 (input box) запускается из этой же формы, без дублирования логики.

## US-20. Дерево сессий не мешает чату — в отдельной панели

**Как** пользователь рана с несколькими сессиями (handoff'ы, compact, …),
**я хочу** видеть дерево сессий в отдельной панели справа, а не поверх ленты сообщений,
**чтобы** чат оставался читаемым на всю ширину, а навигацию по сессиям я открывал тогда, когда она реально нужна.

Acceptance:

- Дерево сессий уезжает из `RunDetails.tsx` в правую collapsible-панель (по аналогии с левой панелью списка ранов).
- В свёрнутом виде — узкая полоска с иконкой «дерево»; в развёрнутом — иерархия сессий с отступами по `parentSessionId`, маркер «●» для активной (live) сессии, бейджи `kind` («🤝 Передача» для agent-agent).
- Состояние свёрнутости — в `globalState`, переживает перезапуск VS Code.
- Клик по сессии — переключает просматриваемую сессию в чате (логика селекта сессии из #0012 сохраняется); композер по-прежнему disabled на неактивной сессии (read-only с подсказкой).
- Если у рана только одна сессия — панель по умолчанию свёрнута и в её шапке отображается «1 сессия» (нет смысла занимать экран).

## US-21. Сообщения в чате читаются как чат

**Как** пользователь,
**я хочу** чтобы сообщения в ленте выглядели как современный мессенджер, а не как лог-файл,
**чтобы** мне было приятно и быстро сканировать историю работы.

Acceptance:

- Каждое сообщение — карточка-bubble с аватаром-иконкой по `from` (user — иконка пользователя; `agent:product`, `agent:architect` — характерные иконки ролей; `agent:system` — нейтральная), именем автора («Вы», «Продакт», «Архитектор», «Система») и временем.
- Время отображается относительно («2 мин назад», «вчера в 14:32») с tooltip'ом, в котором — абсолютная дата/время.
- Сообщения от user'а визуально отличаются от сообщений агентов (выравнивание/фон/акцентный бордер) — глазами видно «моё / не моё».
- Длинные сообщения не ломают вёрстку; markdown в брифе/плане рендерится корректно (заголовки, списки, code-блоки), а не как сырой текст.
- Сообщения подгружаются и обновляются live (как и сейчас через `runs.message.appended`); при новом сообщении лента плавно прокручивается вниз, если пользователь уже был внизу. Если он отскроллился вверх — появляется ненавязчивая пилюля «↓ Новые сообщения».

## US-22. Вызовы инструментов — структурированные карточки, а не строки

**Как** пользователь, наблюдающий за работой агента,
**я хочу** видеть вызовы инструментов как аккуратные карточки со статусом и раскрываемыми деталями,
**чтобы** одним взглядом понимать, что происходит, и при необходимости — без труда заглядывать внутрь.

Acceptance:

- Tool-call в ленте — отдельная компактная карточка с: иконкой инструмента (lucide, маппинг `kb.list` → список, `kb.write` → файл-плюс, `kb.read` → файл-глаз и т.п.), названием тула, статусом (running с пульсацией / ok / error с красным акцентом), однострочным summary (например, `path: "features/dark-theme"`).
- В свёрнутом виде карточка занимает одну строку. По клику — раскрываются полный JSON input, JSON output (или текст ошибки), время выполнения. Подсветка JSON монохромная по vscode-токенам (без сторонних тяжёлых тем).
- Кликабельные пути файлов внутри карточки сохраняют поведение из US-11 (открывают файл в редакторе VS Code).
- Состояние «развёрнут/свёрнут» каждой карточки — локально в компоненте; по умолчанию все карточки свёрнуты, кроме последней с ошибкой (она раскрыта автоматически — ошибки не должны прятаться).
- Tool-карточка визуально отличается от чат-bubble (другая форма, нейтральный фон, моноширинный шрифт для аргументов) — пользователь не путает «сказал агент» и «вызвал инструмент».

## US-23. Всегда видно, что именно сейчас делает система

**Как** пользователь, ждущий результата,
**я хочу** в любой момент видеть осмысленную подпись «что сейчас происходит» вместо немого спиннера или пустого экрана,
**чтобы** не сомневаться, жив ли процесс, и понимать, на каком этапе мы находимся.

Acceptance:

- В шапке рана и на агенте на канвасе (US-25) отображается человеческая строка статуса, выводимая из `meta.status` + последнего активного действия:
  - `draft` → «Готовлюсь к запуску…»
  - `running` без активного tool → «{Роль} думает…» (например, «Архитектор думает…»)
  - `running` с активным tool → «{Роль}: вызов `{tool}`…» (например, «Продакт: чтение kb…»)
  - `awaiting_user_input` → «{Роль} ждёт твой ответ»
  - `awaiting_human` → «Готово — твой ход»
  - `failed` → «Ошибка: {краткое описание}»
- Длительные пустые контейнеры (загрузка списка ранов, открытие деталей) показывают Skeleton-плейсхолдеры, а не пустоту или одиночный спиннер.
- Любая кнопка, инициирующая IPC-запрос (создать ран, отправить сообщение, переключить сессию), на время запроса показывает inline-спиннер с подписью действия и блокируется от повторного клика.
- Если IPC-запрос завис дольше N секунд (для start: ~10с, для send: ~5с) — рядом со спиннером появляется подсказка «Долго отвечает — проверь сеть или ключ» с кнопкой «Отменить».

## US-24. Видеть карту команды агентов на одном экране (canvas)

**Как** пользователь, выбравший ран,
**я хочу** на главном экране рана видеть визуальную карту: кубики-агентов и связи между ними,
**чтобы** одним взглядом понимать, кто участвует, кто кому передавал работу и в каком состоянии команда сейчас находится.

Acceptance:

- При выборе рана main-area по умолчанию открывается на вкладке **«Карта»** (canvas), а не на ленте чата. Чат остаётся доступен второй вкладкой («Чат») — переключение сохраняется per-run в `globalState`.
- Canvas рендерит каждого участника как «кубик» (карточку) с иконкой роли, именем («Продакт», «Архитектор», «Вы»), статусом (US-23) и временем последней активности.
- Между участниками рисуются связи: handoff (`parentSessionId` → bridge) — стрелка от роли-источника к роли-приёмнику с пометкой «передал бриф» / «передал план» (по типу артефакта).
- Layout детерминированный: одинаковый набор сессий → одинаковая раскладка между перезапусками. Допускается простой layered-layout (поколения по `parentSessionId`); полноценный force-directed не нужен.
- Canvas масштабируется (кнопки zoom +/-/100% и колёсико), drag для панорамирования. Состояние zoom/offset — в-памяти на время сессии (без persist).
- На пустом ране (только что создан, агенты ещё не стартовали) canvas показывает ровно одного агента (продакта) и подпись под ним «Запускается…».

## US-25. Видеть, кто из агентов прямо сейчас работает

**Как** пользователь,
**я хочу** на канвасе видеть, какой агент в данный момент активен,
**чтобы** не открывать чат, чтобы это понять.

Acceptance:

- На кубике активного агента вращается компактный спиннер с подписью статуса (из US-23). На неактивных — спиннера нет.
- Кубик агента, ожидающего пользователя (`awaiting_user_input` для его сессии), помечается акцентным бордером и иконкой «вопрос» с подсказкой «Ждёт твоего ответа».
- Кубик агента в `failed`-состоянии помечается красным акцентом и иконкой ошибки.
- Состояние обновляется live — без перезагрузки и без явного обновления — по подписке на `runs.updated`.
- Если agent ничего не делал > N секунд (idle) и при этом ран не закончен — кубик не помечается «работает». «Работает» только когда статус его сессии `running`.

## US-26. Видеть, как агенты передают работу друг другу (анимация)

**Как** пользователь,
**я хочу** в момент, когда один агент передаёт работу другому или пишет ему сообщение в bridge-сессии, видеть на канвасе анимированную стрелку,
**чтобы** ощущать живой обмен между ролями, а не статичную диаграмму.

Acceptance:

- В момент handoff (создаётся bridge-сессия + первое сообщение) между агентом-источником и агентом-приёмником на канвасе появляется анимированная стрелка (pulse / dash-flow в направлении приёмника). Анимация идёт ~3 секунды и затухает, оставляя статичную стрелку.
- Когда в bridge-сессии появляется новое сообщение от роли A к роли B (агент-агент или user → агент при hybrid) — соответствующая стрелка кратко «вспыхивает» (~1 сек), показывая направление передачи.
- Если пользователь вмешался в bridge (US из #0012 hybrid) — на канвасе появляется временная стрелка от user-аватара к адресуемому агенту (видно, что юзер вступил в диалог).
- Анимации не блокируют UI и корректно сворачиваются, если у пользователя включён «prefers-reduced-motion» (заменяются на короткое изменение цвета без движения).
- При входе в ран в середине диалога (открыли давно идущий ран впервые) канвас показывает все стрелки статически, без воспроизведения «истории» анимаций — анимация только для событий, происходящих в реальном времени.

## US-27. В один клик открыть работу любого агента в любой момент

**Как** пользователь,
**я хочу** кликом по кубику агента или по стрелке между агентами проваливаться в их чат,
**чтобы** мгновенно посмотреть, что и в каком виде они говорили друг другу, без поиска нужной сессии руками.

Acceptance:

- Клик по кубику агента → main-area переключается на вкладку «Чат» с выбранной последней (live) сессией этого агента. Если у агента было несколько сессий — открывается самая свежая, дерево сессий справа подсвечивает выбранную.
- Клик по стрелке между двумя агентами → открывается bridge-сессия, которая соответствует этому handoff'у.
- Двойной клик / shift-клик по кубику → открывает popover с компактной хронологией всех сессий этого агента (список с датами и статусами); из него — в любую конкретную сессию.
- Возврат на канвас — кнопка «← Карта» в шапке main-area или клик по вкладке «Карта». Last-viewed сессия в чате запоминается per-run, чтобы повторный заход в «Чат» открывал её же.
- На канвасе при наведении на кубик/стрелку — tooltip с краткими фактами (имя сессии, последнее сообщение, число сообщений, длительность) — drill-in без обязательного перехода.

## US-28. Canvas как иерархия команды, а не flow со стрелочками

**Как** пользователь рана с несколькими ролями,
**я хочу** видеть на canvas простую иерархическую карту команды (кубики участников по уровням), а не flow-граф рабочего процесса со стрелками между всеми коммуникациями,
**чтобы** одним взглядом понимать состав команды и кто кому подчиняется, не теряясь в переплетении линий по мере того как агенты общаются всё больше.

Контекст (важный — это и есть концептуальный сдвиг):

- Изначальная идея была — отображать весь рабочий процесс и все рабочие коммуникации в виде flow со стрелочками.
- Эта идея отвергнута, потому что (а) её слишком сложно реализовать: участники разработки могут коммуницировать друг с другом много раз и все эти связи будут переплетаться; (б) такой вариант теряет часть информации — «кто когда и с кем взаимодействовал».
- Вместо flow на canvas показываем **просто кубики участников разработки** в иерархии. Реальная история работы будет жить отдельно — хронологическим списком чатов (US-29).

Acceptance:

- На canvas в виде кубиков показаны участники разработки: продакт, архитектор, разработчик — и другие роли в будущем по мере появления.
- Кубики расположены в виде **иерархии**: пользователь — самый главный (наверху), потом продакт, потом архитектор, потом программист.
- Иерархия предусматривает несколько программистов в будущем (одного уровня под архитектором). Сейчас — один.
- Никаких стрелок-flow, изображающих последовательность работы или историю передач, на canvas нет. Между уровнями допустимы только статичные тонкие линии «репортинга» (org-chart-стиль), которые не растут с числом коммуникаций.
- При появлении новой роли в проекте (например, дизайнера) добавление в иерархию — изменение одной константы, не редизайн canvas.

## US-29. Хронологический список встреч с участниками (журнал работы команды)

**Как** пользователь, наблюдающий за работой команды агентов,
**я хочу** видеть реальную историю работы как хронологический список «джоб или встреч» с указанием участников каждого чата,
**чтобы** в любой момент видеть «кто когда и с кем взаимодействовал» — то, что flow-диаграмма теряла.

Контекст: «А саму их работу будем логировать в виде хронологического списка джоб или встреч с участниками (по сути чаты агентов).»

Acceptance:

- Список «встреч/джоб» показывается отдельным видом в UI (рядом с canvas или вкладкой). Каждая запись = чат-сессия рана.
- Каждая запись отображает: участников чата, инициирующий вход (кто/что вызвало этот чат), время старта, статус.
- **Важное правило про участников:** автор входного артефакта **не считается** участником чата, если он только передал данные и не вёл диалог. Например, продакт сделал brief и передал архитектору — продакт **не** участник этой записи; архитектор работает один. В записи пишем «input поступил от продакта, и именно это инициировало работу архитектора».
- Сценарии (закрепить как TC):
  1. **User → Product (старт рана):** создаётся начальный чат между пользователем и продакт-агентом. Запись добавляется в хронологический список, в ней **2 участника**: пользователь и продакт-агент.
  2. **Product → Architect (передача brief):** новая запись в списке. Это передача информации, поэтому в чате работает только архитектор один. **Участник — только архитектор** (пользователь — implicit, как заказчик; продакт — **не** участник, он source of input). В записи: «input от продакта инициировал работу архитектора».
  3. **Architect → Programmer (передача plan):** отдельная запись. Работа программиста. **Участник — только программист.** Input от архитектора.
  4. **Programmer → Product (вопрос «снизу»):** новая запись. Программист зовёт продакта в чат, и **обязательно** должен также позвать архитектора (см. US-31). Участники: программист, архитектор, продакт.
  5. **Architect → Product:** отдельная запись. Архитектор зовёт продакта. Если они на соседних уровнях, дополнительно никого звать не надо (см. US-31). Участники: архитектор, продакт. Допустимо: архитектор и программист **вместе** зовут продакта в чат — тогда продакт сразу получает контекст и программист сразу получит ответ; участники: программист, архитектор, продакт.
- Список — реальная история, в любой момент полный и точный.

## US-30. Многоучастниковые чаты (комнаты) и группировка на canvas

**Как** пользователь, наблюдающий за общением команды,
**я хочу** что когда несколько агентов общаются в одном чате, они визуально группировались как бы в отдельной комнате на canvas, а коммуникации между двумя агентами показывались анимацией между их кубиками,
**чтобы** ощущать живой процесс общения, не превращая canvas в граф-кашу.

Контекст: «не надо на canvas-е показывать сложные цепочки или последовательности. Просто показываем иерархию команды и если кто-то с кем-то общается, то надо между ними показать анимацию общения. Если несколько участников общаются в чате, то они должны группироваться как бы в отдельной комнате.»

Acceptance:

- Когда два агента общаются в активном чате — между их кубиками идёт анимация коммуникации (pulse / dash-flow). Стрелка не «копится» статически — она существует только пока чат активен.
- Когда в чате 3+ участников — их кубики визуально группируются «в одной комнате» (например, выделяющим контейнером/ореолом, объединяющим их). Группировка появляется при создании комнаты и исчезает при её закрытии.
- Чаты с N>2 участниками — first-class сущность, а не набор парных стрелок. Один pull-in роли в чат не плодит дополнительных линий, а расширяет границу комнаты.
- Группировка не нарушает иерархическую раскладку (US-28): кубики остаются на своих уровнях, группа «обнимает» их там, где они стоят, без перетасовки.

## US-31. Правило «через уровень нельзя» при кросс-уровневых обращениях

**Как** пользователь, поддерживающий аналогию с реальной командой,
**я хочу** что если агент обращается к коллеге через уровень иерархии, то промежуточные уровни автоматически попадали в этот же чат участниками,
**чтобы** работа строилась как в реальной команде — junior не идёт через голову тимлида к продакту в одиночку.

Контекст:

- «если программист зовёт продакта, то он также должен позвать агента (или агентов) между этими уровнями (т.е. своего начальника). В данный момент только архитектора.»
- «Если вопрос к архитектору (т.е. между участниками нет уровней), то дополнительно никого звать не надо.»
- «Если у архитектора возник вопрос к продакту, то архитектор и программист могут просто позвать продакта в чат. Тогда продакт сразу получит необходимый контекст.» — то есть приглашение «соседей по уровню сверху и снизу» допустимо как удобство, чтобы они сразу получили ответ.

Acceptance:

- При обращении роли A к роли B через уровень иерархии все промежуточные уровни **обязательно** добавляются в чат как полноценные участники.
- Сейчас единственный кросс-уровневый кейс — программист → продакт, между ними промежуточный уровень архитектор. Чат получает участников `[programmer, architect, product]`.
- При обращении к соседнему уровню (например, архитектор → продакт, или программист → архитектор) дополнительные участники **не нужны**.
- Допустимо «подтянуть» в чат соседа по уровню сверху или снизу, чтобы он сразу получил ответ (например, архитектор зовёт продакта и берёт с собой программиста — программист сразу услышит ответ продакта). Это не обязательно, но разрешено.
- Правило выводимо из иерархии (US-28): порядок ролей задан, промежуточные между A и B вычисляются однозначно.

## US-32. Запрос на встречу и пауза, когда участник занят

**Как** пользователь, наблюдающий за работой команды,
**я хочу** что если кто-то из агентов занят в момент обращения к нему, создавался запрос на встречу, а инициирующий агент ставился на паузу до получения ответа,
**чтобы** работа строилась как в реальной команде, без «телепортации» занятого коллеги в твой чат.

Контекст:

- «Если кто-то из участников занят, то создается запрос на встречу в чате и агенты, который нужен ответ становятся на паузу. Потому что их работе нужен ответ.»
- «Если агент на паузе (т.е. он ожидает доступности), то его можно позвать на встречу.»

Acceptance:

- Когда агент A пытается начать чат с агентом B (или вытащить B в существующую комнату), и B сейчас занят — система создаёт **запрос на встречу** в этом чате.
- Агенты, которым нужен ответ, переходят в состояние **«пауза»** — они не делают новых шагов, потому что их работе нужен ответ от B.
- Когда B освобождается — запрос на встречу резолвится, B попадает в чат и может ответить.
- Агента, находящегося на паузе (то есть ожидающего доступности другого участника), можно позвать на встречу. Он перестаёт быть «недоступным» в этом смысле — его пауза не блокирует приглашения от других.
- На canvas пауза агента видна (отдельным состоянием на кубике, отличимым от idle и working).
- Состояние «пауза» переживает рестарт VS Code — после перезапуска ран продолжает корректно ждать освобождения нужного участника.

## US-33. Навигация по журналу встреч через prev/next связи

**Как** пользователь, разбирающий историю рана,
**я хочу** чтобы у каждого чата были связи **prev** (что инициировало этот чат) и **next** (что появилось после), причём каждая связь может быть массивом,
**чтобы** я мог переходить из одной встречи в породившие её и в порождённые ею, и при ветвлении выбирать, в какую идти.

Контекст: «При этом у чата может быть prev и next параметры. prev — джоба, которая инициировала этот чат. next — джоба или чат, который появился после текущего чата. prev и next может быть массивом. Тогда можно выбирать в какой перейти.»

Acceptance:

- Каждая запись в хронологическом списке (US-29) содержит поля `prev` и `next`, каждое — массив ссылок на другие записи (чаты/джобы).
- `prev` — те джобы, которые инициировали этот чат (обычно одна, но допустимо несколько — например, если чат собрал контекст из нескольких источников).
- `next` — джобы/чаты, которые появились после этого чата как его следствие (одна или несколько — один чат может породить несколько последующих, например архитектор после встречи с продактом инициирует и работу программиста, и параллельную запись в kb).
- В UI карточки чата `prev` и `next` показаны как кликабельные ссылки. При наличии нескольких — пользователь видит список и **выбирает, в какую перейти**.
- Граф prev/next цикличен быть не должен (это история, а не граф зависимостей в общем смысле); UI не обязан явно ловить циклы, но и не должен на них ломаться.
- prev/next — не «дерево родителей-сессий» в текущем технического смысле, а явная семантика «откуда родилось / что породило». В частности, передача артефакта (brief, plan) как input — это связь prev/next, даже если автор артефакта не участник чата.

## US-35. (инфраструктурно) Метаданные встречи на сессии: inputFrom + prev/next

Задача #0035 — типовая инфраструктура под журнал встреч (#0029 → #0046):
US-33 уже зафиксировал требование «у чата есть prev и next», но storage
до этой задачи знал только про `parentSessionId` (один родитель,
одиночное поле). Прямой пользовательской истории не порождает —
закрепляет контракт типа.

Acceptance (закрепление контракта типа, проверяется тестами и кодом):

- `SessionMeta` несёт три новых поля: `inputFrom: Role | 'user'` —
  кто/что инициировало сессию (роль автора входного артефакта или
  `'user'`), `prev: SessionId[]` — родительские сессии, `next: SessionId[]` —
  дочерние.
- При создании дочерней сессии `prev` проставляется явно, `next`
  родителя дописывается одним write+persist (и в session-meta родителя,
  и в его summary в run-meta).
- Read-time normalization для legacy session-meta: `prev` восстанавливается
  из `parentSessionId`, `next` пересчитывается обратным индексом по всем
  сессиям рана (один проход в `readMeta`/`listAllMeta`), `inputFrom` —
  по `participants[0]` родителя, `'user'` как фолбэк.
- `parentSessionId` остаётся как алиас на `prev[0]` для обратной
  совместимости (удаление — отдельная задача в будущем).

## US-37. Агент зовёт соседа по иерархии через team.invite

**Как** агент в активном ране (продакт / архитектор / программист),
**я хочу** позвать соседнего по иерархии агента в текущую сессию-комнату с
сопроводительным сообщением,
**чтобы** обсудить с ним вопрос или передать задачу без эскалации через
посредников.

Прямого UI-сценария на этом этапе нет — приглашение делает модель внутри
agent-loop'а, вызывая тул `team.invite(targetRole, message)`. Пользователь
видит результат как обычно: новое сообщение в чате от роли-инициатора и
обновлённый состав участников комнаты (после рендера в #0046).

Acceptance:

- Тул `team.invite` зарегистрирован в реестрах продакта, архитектора и
  программиста; для каждой роли caller жёстко зашит в фабрику —
  модель не может «представиться» другой ролью через аргументы.
- Соседи по иерархии: `product ↔ architect`, `architect ↔ programmer`.
  Через уровень (`programmer → product`, `product → programmer`) и
  «сам себя» (`architect → architect`) тул отказывает с текстом
  «Нельзя пригласить через уровень. Используй team.escalate(targetRole,
  message)» — это сигнал модели, что есть правильный путь (#0038).
- При успехе: вызывается `pullIntoRoom(currentSessionId, targetRole)`
  (#0036), затем сообщение `message` от имени caller'а пишется в
  `chat.jsonl` той же сессии. Тул возвращает `{sessionId, participants}`
  с актуальным составом комнаты.
- Идемпотентность повторного invite той же роли: участник не
  дублируется, событие `participant_joined` не дублируется (это
  гарантирует pullIntoRoom), но новое сообщение в чат пишется — это
  корректно, повторный invite — это просто новый месседж в комнату.
- Без интеграции с meeting-request (#0051): если target busy — всё
  равно тащим в комнату. Очередь встреч появится отдельной задачей.

## US-38. Агент эскалирует через уровни иерархии через team.escalate

**Как** агент в активном ране (программист или продакт),
**я хочу** позвать роль через уровень с автоматическим затаскиванием
посредников,
**чтобы** не нарушать правило команды «между программистом и продактом
обязательно архитектор» и не учить модель собирать цепочку вручную.

Прямого UI-сценария нет — эскалацию делает модель внутри agent-loop'а,
вызывая тул `team.escalate(targetRole, message)`. Пользователь видит
результат как обычный чат: новое сообщение от роли-инициатора в
сессии-комнате и обновлённый состав участников (рендер — после #0046).

Acceptance:

- Тул `team.escalate` зарегистрирован в реестрах продакта, архитектора и
  программиста; для каждой роли caller жёстко зашит в фабрику —
  модель не может «представиться» другой ролью через аргументы.
- Цепочка собирается ОДИН раз до первого `pullIntoRoom`:
  `[caller, ...rolesBetween(caller, target), target]`. Для
  `programmer → product` цепочка — `[programmer, architect, product]`,
  для `product → programmer` — `[product, architect, programmer]`
  (escalate работает в обе стороны).
- Для каждой роли цепочки кроме caller'а вызывается
  `pullIntoRoom(currentSessionId, role)` (#0036). pullIntoRoom
  идемпотентен — если посредник уже в комнате, событие
  `participant_joined` не дублируется.
- Сообщение `message` от имени caller'а пишется в `chat.jsonl` сессии
  ОДИН раз ПОСЛЕ всех `pullIntoRoom` — чтобы у всех приглашённых был
  одинаковый стартовый контекст (полная комната с момента входа).
- Тул возвращает `{sessionId, participants, chain}` — текущий состав
  комнаты + полная цепочка эскалации.
- Если `caller === target` или `areAdjacent(caller, target)` (соседний
  уровень) — escalate отказывает с текстом «escalate не нужен,
  используй team.invite или прямой ответ»: это сигнал модели, что есть
  правильный путь (`team.invite` или прямой ответ в чат).
- Без интеграции с meeting-request (#0051): если кто-то из посредников
  busy — всё равно тащим в комнату.
- Допустимое частичное состояние: если `pullIntoRoom` посредине упал —
  комната может остаться без последующих участников, без сообщения
  caller'а. Диагностика и компенсация — будущей задачей; сейчас не
  усложняем тул двухфазным коммитом ради редкого сбоя.

## US-36. (инфраструктурно) pullIntoRoom + системное событие participant_joined

Задача #0036 — базовый кирпич для будущих тулов `team.invite` (#0037) и
`team.escalate` (#0038): action `pullIntoRoom(sessionId, role)` добавляет
агентскую роль в `participants` сессии и пишет в журнал сессии системное
событие `participant_joined`. Прямого пользовательского сценария не
порождает (рендера в UI пока нет — он появится в #0046 «Журнал встреч»),
закрепляет контракт API и формат события.

Acceptance (закрепление контракта API, проверяется тестами и кодом):

- `pullIntoRoom(runId, sessionId, role)` идемпотентен: если роль уже среди
  `participants` — no-op (`undefined` на выходе, ни записи участника, ни
  события в журнале).
- При реальном добавлении: участник появляется в `participants` (и в
  `SessionMeta`, и в `RunMeta.sessions[].participants`); в `tools.jsonl`
  сессии пишется ровно одна запись `{kind: 'participant_joined', at, role}`.
- Тип `participant_joined` присутствует в дискриминированном union'е
  `ToolEvent` и зеркалится в webview-копии типа.
- Запись события идёт в `tools.jsonl` (а не в `chat.jsonl`) — это
  системная запись, рендер которой задумывается отдельным стилем; в текущей
  ленте `RunDetails` запись скрывается, чтобы не шуметь рядом с обычными
  сообщениями. Полноценный рендер делается в #0046.
- Возвращённая `RunMeta` готова к broadcast'у `runs.updated` без
  дополнительного `readMeta` — подписчики live-state увидят новый состав
  `participants` сразу после вызова (через каллер; тесты проверяют само
  возвращаемое значение и состояние диска).

## US-34. (инфраструктурно) Сессия как комната: participants — массив произвольной длины

Задача #0034 — чисто типовая миграция, прямого пользовательского сценария
не порождает: формат `Session.participants` меняется с «жёсткой пары»
на массив длины ≥ 1, чтобы #0036 (`pullIntoRoom`) и #0038 (`escalate`)
могли создавать чаты на N>2 участников без последующих ломок storage.

Acceptance (закрепление контракта типа, проверяется тестами и кодом):

- `SessionMeta.participants: Participant[]` всегда непустой массив.
- При чтении старых session-meta без массива (legacy `agentRole` / только
  `kind`) extension восстанавливает участников в массив длины 2 в памяти,
  файл на диске не переписывает.
- Новые сессии всегда сериализуются в новом формате, длина 2 и 3 — round-trip
  без потерь.
- Никаких `participants[0]/participants[1]/length === 2` в коде; обход —
  через `some/filter/map`.

## US-39. Промпты ролей: правило escalate vs invite

**Как** инженер, поддерживающий промпты ролей,
**я хочу** чтобы во всех системных промптах было однозначное правило, когда
применять `team.invite`, а когда `team.escalate`,
**чтобы** агенты не звали продакта в обход архитектора и не дублировали
ручную сборку цепочки посредников.

Прямого UI-сценария нет — это контракт системного промпта (тулы #0037 и
#0038 уже работают, но без soft-rule в промпте модель всё равно может
выбрать неправильный путь). Проверяется снапшот-стайл подстроками в
unit-тестах + ручным TC.

Acceptance:

- В каждом из трёх промптов (`product.prompt.ts`, `architect.prompt.ts`,
  `programmer.prompt.ts`) есть раздел `## Команда и эскалация` с описанием
  иерархии `User → product → architect → programmer` и правилом «соседний
  уровень → `team.invite`, через уровень → `team.escalate`».
- В промпте программиста есть конкретный пример: вопрос продакту →
  `team.escalate('product', '<вопрос>')` → итоговый состав комнаты
  `[programmer, architect, product]`.
- В промпте архитектора зафиксировано, что `team.invite` соседей
  (`product`/`programmer`) — норма, а `team.escalate` ему практически не
  нужен (через уровень от архитектора нет).
- В промпте продакта зафиксировано, что `team.escalate('programmer', ...)`
  автоматически подтянет архитектора в комнату.
- Никаких изменений в коде вне промпт-файлов (#0039: задача чисто на
  правила в тексте промптов).
