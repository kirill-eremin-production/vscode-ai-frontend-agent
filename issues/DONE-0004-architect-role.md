## Context

Архитектор — следующий шаг цепочки `продакт → архитектор → планировщик → программисты`. На вход получает `brief.md` от продакта (#0003), на выход выдаёт `plan.md` — технический план: разбиение на подзадачи, архитектурные решения, риски.

Как и продакт, архитектор — не stub. У него:

- собственная kb (`.agents/knowledge/architect/`) с накопленной памятью об архитектуре проекта (модули, паттерны, прошлые решения, отвергнутые альтернативы);
- доступ к `ask_user` для уточнений, которые невозможно вывести из брифа (продакт отвечает за продуктовую неопределённость, архитектор — за техническую);
- по-прежнему **нет** доступа к коду проекта на этой итерации. Tool runtime для чтения произвольных файлов рабочего проекта — отдельная будущая задача (нужна для программиста, и архитектор тогда тоже её получит). Пока работает по брифу + своей kb.

## Acceptance criteria

- Архитектор стартует автоматически после успеха продакта (`awaiting_human` с готовым `brief.md` → `running` архитектора). На текущей итерации без явного approve пользователя; кнопки approve/reject между ролями — отдельная будущая задача.
- Используется **более «умная» модель**, чем у продакта; имя константой в `src/extension/entities/run/roles/architect.ts`.
- Структура kb архитектора (зеркальная #0002, под свою предметную область):
  - `modules/` — описания модулей/слоёв проекта.
  - `decisions/` — архитектурные решения (ADR-style).
  - `patterns/` — устоявшиеся паттерны (как делаем IPC, как пишем тесты, и т.п.).
  - `risks/` — известные риски и грабли.
  - `README.md` — описание схемы (как у продакта).
- System prompt описывает роль, формат `plan.md`, kb-конвенцию, обязательный паттерн «сначала прочитать релевантные разделы своей kb по областям из brief.md → потом думать → задать вопросы при пробелах → записать новые decisions/patterns в kb → выдать plan».
- Доступные тулы: `kb.read`, `kb.write`, `kb.list`, `kb.grep` (sandbox `.agents/knowledge/architect/`), `ask_user`. **Не разрешаем** читать `knowledge/product/` — связь между ролями только через артефакты ранов (`brief.md`). Если архитектору не хватает продуктового контекста — он задаёт `ask_user`, не лезет в чужую kb.
- Архитектор читает `.agents/runs/<id>/brief.md` через специальный тул `run.readBrief()` (или просто получает текст брифа в начальном user-message — финализируем при реализации; склоняюсь ко второму, т.к. brief всегда один и нужен сразу).
- Финальный `plan.md` сохраняется атомарно. Жёсткий формат:
  - `# План` + аннотация (1–2 предложения).
  - `## Цели` — технический перевод acceptance из брифа.
  - `## Подзадачи` — нумерованный список. Для каждой: `### N. <название>`, описание, затрагиваемые модули (на уровне путей-паттернов вроде `src/extension/...`, без чтения кода), зависимости от других подзадач.
  - `## Архитектурные решения` — что выбрали и почему, что отбросили.
  - `## Риски и граничные случаи`.
  - `## Связанные артефакты kb` — пути в `knowledge/architect/`, на которые опирался или которые создал/обновил.
- Сообщения в `chat.jsonl` (от `agent:architect`, `ask_user` пары) и tool-лог в `tools.jsonl` — по той же схеме, что у продакта.
- Статусы: `awaiting_human` (после продакта) → `running` (архитектор) → `awaiting_user_input` на `ask_user` → `awaiting_human` (план готов) | `failed`.
- В UI секция `RunDetails` показывает оба артефакта — `brief.md` и `plan.md` — последовательно.
- Ошибки (нет `brief.md`, сеть, пустой ответ, превышен лимит итераций) — `failed` с диагностикой.

## Implementation notes

- `runArchitect(runId, context)` живёт в `service.ts`, вызывается по success-пути из `runProduct`. Если хочется явного оркестратора — заведём при реализации, заранее не выделяем.
- Передавать brief в архитектора как user-message (полный текст). Парсить секции брифа на этой итерации не нужно — модель справится сама. Парсер появится только если конкретно понадобится скармливать архитектору только часть брифа (например, при повторном проходе).
- TC писать **до реализации**: успешный ран сквозь продакта и архитектора, отсутствие `brief.md` (архитектор стартует руками без продакта — диагностика), сетевой сбой архитектора, ран с `ask_user` от архитектора, перерисовка UI после готовности плана.
