## Context

Канвас из #0023 — статичный. Чтобы он реально работал как «офис на ладони», на каждом кубике должно быть видно прямо сейчас: думает агент или ждёт пользователя, упал ли, или просто idle. Без этого канвас остаётся «диаграммой архитектуры» — красивой, но мёртвой.

В этой задаче кубики оживают: на работающем агенте крутится спиннер с подписью, на агенте, ждущем юзера — акцентный бордер и иконка вопроса, на упавшем — красный акцент. Всё реактивно, без перезагрузки.

## Acceptance criteria

- На каждом кубике агента отображается компонент `LoadingState` (#0016) с подписью из `describeRunActivity` (#0022), вычисленной для данной роли:
  - `idle` → нейтральный, без спиннера, подпись «—» или последняя осмысленная (например, «закончил {brief|plan}»).
  - `thinking` → крутится спиннер, подпись «Думает…».
  - `tool` → спиннер с альтернативной иконкой (lucide `Wrench` пульсирует), подпись «Вызов `{tool}`…».
  - `awaiting_user` → нет спиннера, акцентный (warning) бордер кубика + иконка «вопрос» в углу + подпись «Ждёт твой ответ».
  - `awaiting_human` → нейтральный бордер, бейдж «Готово».
  - `failed` → красный бордер + иконка ошибки + подпись «Ошибка» (полный текст в tooltip'е).
- Ширина кубика динамически вмещает подпись (с разумным max-width, длинные подписи — truncate с tooltip'ом). Высота фиксированная.
- Время последней активности на кубике обновляется live (`formatRelativeTime` из #0020 + `setInterval(60s)` для перерасчёта «N мин назад»). Достаточно одного интервала на канвас, не на каждый кубик.
- Подсветка обновляется реактивно по подписке на `runs.updated`, `runs.message.appended`, `runs.tool.appended` — без вмешательства пользователя.
- При смене активной сессии в ране (handoff) фокус «активной роли» переезжает на нового агента: старый кубик становится `idle` (или «закончил»), новый — `thinking`/`tool`. Это естественно следует из `describeRunActivity`, но проверить, что при handoff'е нет «двух одновременно работающих» кубиков (когда продакт уже завершился, а архитектор только начал — должен быть короткий момент `idle` у обоих, без визуального бага).
- На user-кубике (если есть в hybrid'е): в `awaiting_user_input` — он подсвечен (как «слово за тобой»), в остальных случаях — нейтральный.
- Performance: `describeRunActivity` пересчитывается на каждом обновлении store — но только для видимого рана (не для всех в списке). Ничего тяжёлого там нет, но проверить, что нет лишних ререндеров (`React.memo` на кубике + стабильные props).

## Implementation notes

- `describeRunActivity` уже работает на per-сессию + per-роль (#0022). На канвасе каждая роль может иметь несколько сессий — берём ту сессию, в которой роль _активна_ (последняя по `updatedAt`, в которой роль есть в `participants` и сессия в `running`/`awaiting_*`). Если такой нет — берём самую свежую закрытую/idle, статус `idle` с подписью «закончил {artifact}».
- Маленький утилит `selectActiveSessionForRole(meta, role): SessionMeta | undefined` — чистая функция, юнит-тестируется.
- Иконку «работает» удобно поставить на угол кубика (corner badge), а строку статуса — внутри кубика под именем. Так визуально не перегружено.
- Анимация спиннера — Tailwind/CSS, без JS-ticker'а. Pulse — те же `animate-pulse` / `animate-spin`. `prefers-reduced-motion` — отключает анимации, оставляет статичную точку.
- TC: расширить e2e — во время первого шага продакта на канвасе у его кубика виден спиннер; после `awaiting_human` — спиннера нет, бейдж «Готово»; после handoff'а спиннер на архитекторе.
