## Context

После #0023/#0024 канвас живой, но _передачи_ работы между агентами видны статически — стрелка появилась и стоит. Чтобы по-настоящему ощущать «команду в работе», нужно показывать момент общения: handoff загорается, новое сообщение в bridge — короткая вспышка вдоль стрелки, user вмешался — временная стрелка от user-кубика.

Этот тикет — про motion и реактив. Никаких новых данных: всё событиями уже приходит (`runs.message.appended` с `sessionId`).

## Acceptance criteria

- **Handoff появление**: при создании bridge-сессии (`runs.updated` с новым элементом в `meta.sessions`, `kind === 'agent-agent'`):
  - Соответствующая стрелка появляется с короткой анимацией: из источника растёт линия, в момент достижения приёмника — наконечник, затем pulse ~3 секунды (dash-flow в направлении приёмника). После — становится статичной.
- **Новое сообщение в bridge**: `runs.message.appended` для bridge-сессии с `from` ∈ {agent:role}:
  - Стрелка от роли-автора к роли-приёмнику (другому участнику bridge'а) даёт короткую вспышку (~1с): подсветка цветом, лёгкий pulse. После — возвращается в обычное состояние.
- **User intervention в bridge** (hybrid): первый раз, когда `participants` bridge'а получает `{kind:'user'}`:
  - На канвасе появляется временная стрелка от user-кубика к bridge'у (точнее — к роли, которая активна в bridge'е сейчас). Стрелка остаётся постоянной (как и user-edge из #0023), а в момент появления — даёт ту же 3-секундную анимацию, как handoff.
  - Каждое последующее `runs.message.appended` с `from === 'user'` в bridge — короткая вспышка той же стрелки.
- **Сообщения в чатах user↔agent**: для обычной user-agent сессии — вспышка двусторонней стрелки (или короткая стрелка от автора к собеседнику, исчезающая через ~1с) между user-кубиком и кубиком агента. Это менее критично, чем bridge-анимация, но желательно для целостности впечатления.
- Анимации не блокируют UI и не мешают взаимодействию (клики по кубикам работают во время анимации).
- `prefers-reduced-motion: reduce` отключает движение, оставляя только короткое изменение цвета (~500мс fade-in/fade-out).
- Историческая «глухота»: при первом открытии канваса для давно идущего рана — все стрелки сразу в статике, никаких воспроизведений прошлых анимаций. Анимация только для событий, **пришедших после монтирования** канваса (флаг `hasMounted` + игнор первого batch'а сообщений).
- Если событий приходит каскадом (типа модель быстро прислала 5 tool-вызовов подряд) — стрелка не «дёргается» 5 раз; debounce/throttle на ~300мс, в течение которого выдаём одну вспышку. Это снимает визуальный шум.

## Implementation notes

- Анимации — CSS keyframes на svg-элементах (path с `stroke-dasharray` + анимация `stroke-dashoffset` для dash-flow). Не подключать GSAP/framer-motion ради этого — overkill.
- Жизненный цикл одной анимации — управляется маленьким хуком `useArrowFlash(edgeId, deps[])`: при изменении deps выставляет `flashing: true` + `setTimeout` снимает через N мс. Edge получает флаг через props и рендерит соответствующий класс.
- Триггеры — подписка в store на новые сообщения. Селектор: `selectLastMessageInBridge(runId, sessionId)`. При изменении его id — флэшим стрелку.
- Throttle/debounce — простой `useRef` с timestamp последнего флэша на конкретный edge.
- Для user-кубика на hybrid'е может не быть детерминированного `(x, y)` пока: добавить его в layout #0023 как опциональный (если в любом `participants` есть `{kind:'user'}` и при этом ран имеет хотя бы одну bridge-сессию — добавляем user-кубик в самый левый столбец, стрелка из него — к bridge'у). Если правка layout'а получится больше пары строк — выделить под неё мини-issue, а в #0025 сделать допущение «user-кубик уже есть».
- TC: e2e — после handoff'а в DOM появляется элемент с `data-arrow-flashing="true"` (или класс), и через 3+ секунды флаг снимается. Реальную «красоту» движения e2e не проверяет — только факт триггера.
