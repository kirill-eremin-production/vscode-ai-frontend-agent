## Context

Это **ядро продукта**: пользователь хочет видеть команду агентов как офис на ладони — карточки-агенты на канвасе с визуальными связями между ними. Чат и дерево сессий — drill-in от этой карты, а не первичный экран.

Эта задача — фундамент: статичный канвас без анимаций и live-индикаторов. Он рендерит агентов из `meta.sessions` и связи (handoff'ы) из `parentSessionId`. Live (#0024), анимация (#0025) и drill-in (#0026) — следующими тикетами.

Ключевая модель: один агент (роль) на канвасе = один кубик. У роли может быть несколько сессий; на канвасе он остаётся одним кубиком — сессии видны как drill-in (#0026), не как разные кубики.

## Acceptance criteria

- Main-area выбранного рана получает вкладки сверху: **«Карта»** (canvas, default) и **«Чат»** (текущий чат-вид). Переключение per-run сохраняется в `globalState` (через `state.setUiPref` из #0017).
- Новый компонент `src/webview/features/canvas/ui/RunCanvas.tsx`:
  - Извлекает из `meta.sessions` уникальный набор ролей-агентов (`product`, `architect`, … + `user` если присутствует в `participants` хотя бы одной сессии).
  - Рендерит каждого агента как карточку (кубик) с `Avatar` (#0016) роли, именем («Продакт», «Архитектор», «Вы»), статичной строкой «idle» (живая активность придёт в #0024) и временем последней активности из `updatedAt` сессий этой роли.
  - Связи (рёбра): для каждой `agent-agent` сессии с `parentSessionId` рисуется стрелка от роли-источника (роль, которая владела родительской сессией) к роли-приёмнику (новая роль в `participants` bridge'а). Подпись на ребре — тип артефакта передачи («бриф», «план», …) — выводится из `RunMeta` (`briefPath`/`planPath`) или, проще, из роли-источника (продакт → бриф, архитектор → план).
  - Если `user` присутствует в `participants` bridge'а (hybrid) — добавляется отдельная стрелка от user-кубика к ближайшему агенту bridge'а; статическая (анимация — в #0025).
- Layout — детерминированный layered:
  - Корни (роли без входящих рёбер) — в первом столбце, отсортированы по первому появлению в `meta.sessions`.
  - Каждое ребро смещает приёмник на один столбец вправо относительно источника.
  - Внутри столбца — вертикально по порядку появления.
  - Без library: руками посчитать `(col, row)` → `(x, y)` через константы (например, 240×140 шаг). Force-directed/dagre/elk пока **не нужны** — у нас 2–4 агента в реалистичном сценарии.
- Канвас — обычный `<svg>` (или div'ы абсолютным позиционированием для кубиков + svg для рёбер). Поддерживает:
  - Колёсико → zoom (с центром в курсоре), кнопки `+ / - / 100%` в углу.
  - Drag по фону → панорамирование.
  - Клавиша `0` или кнопка «100%» → сброс zoom/offset.
  - Состояние zoom/offset — in-memory на время жизни вкладки, без persist (для MVP).
- Если у рана только что создан и сессия одна — канвас показывает один кубик (продакта) с подписью статуса (`describeRunActivity` из #0022 даст «Готовлюсь к запуску…» / «Продакт думает…»).
- Стрелки — простые ломаные/Безье от правой грани источника к левой грани приёмника. Стрелка имеет наконечник и подпись над серединой.
- Канвас респонсивный: при изменении ширины main-area переcчитывает viewBox без сброса zoom/offset.
- Ошибки/edge-cases:
  - Если рендер падает — показать fallback-баннер «Карта временно недоступна, открой вкладку Чат» + ссылка на чат. Не валить весь webview.

## Implementation notes

- Функция layout'а — чистая, тестируема: вход = `RunMeta.sessions`, выход = массив `{role, x, y}` + массив рёбер `{from, to, label}`. Завести юнит-тест на 3 кейса: 1 агент, продакт→архитектор, продакт→архитектор+user (hybrid).
- Не подключать `react-flow` / `xyflow` пока — они дают много, но добавляют ~200KB в bundle и свой ментальный оверхед. Для текущей сложности ручной svg покрывает все требования. Если вдруг сложность вырастет (10+ агентов, draggable layout) — заводим отдельный тикет на миграцию.
- Координаты внутри svg удобнее держать в `viewBox`-пространстве; zoom/pan — через изменение `viewBox` (а не CSS-transform), чтобы стрелки и текст оставались чёткими.
- Кубики — внешне как `Panel` (#0016) с цветной полосой акцента по роли (`var(--color-role-product)` и т.п.).
- TC: новый e2e-смок: после старта рана и handoff'а на канвасе видно два кубика (Продакт, Архитектор) и одну стрелку с подписью «бриф». Проверка через DOM-querySelector на data-атрибутах (`data-canvas-role="product"`, `data-canvas-edge="product->architect"`).
