## Context

Текущий canvas (#0023–#0026) пытается показывать рабочий процесс как flow со стрелками между ролями. Это работает на простой цепочке product → architect → programmer, но ломается, как только участники начинают общаться многократно и не по линии: стрелок становится много, они переплетаются, и хуже того — flow-картинка не отвечает на вопросы «кто сейчас занят», «кто с кем разговаривает прямо сейчас». Историю взаимодействий (кто/когда/с кем) flow тоже теряет — стрелки агрегируют, а не упорядочивают.

Решение — развести две вещи:

- **Canvas = иерархия команды (org-chart)**, статичная картинка «кто кому подчиняется», плюс live-индикаторы состояния каждого участника. Не растёт с числом коммуникаций.
- **Хронологический список чатов** (отдельная задача #0029) — реальная история «кто когда с кем встречался».

Эта задача — про первую часть: убрать flow-лэйаут, поставить иерархию.

## Acceptance criteria

**Лэйаут:**

- Уровни сверху вниз: User → Product → Architect → Programmer. Один кубик на роль (на этой итерации; параллельные программисты — будущее).
- User — отдельный визуальный элемент сверху (аватар или иконка), не «такой же кубик» как агенты. Это подчёркивает, что user — заказчик, а не участник иерархии агентов.
- Кубики не перетаскиваются (на первой итерации). Позиции детерминированы по роли и уровню.
- Между уровнями — тонкие статичные линии-«репортинги» (не путать со стрелками коммуникации). Это org-chart, а не flow.

**Состояние участника на кубике:**

- `idle` — серый/нейтральный.
- `working` — пульсирующий индикатор + (если возможно) короткое описание текущего действия (например, имя текущего тула).
- `paused` — индикатор «ожидает ответа» (например, ждёт ответа от другого участника — см. #0031).
- `awaiting_user` — индикатор «требуется ответ пользователя» (как сейчас).

Состояние выводится из активных сессий рана и статуса соответствующего ран-файла (как сейчас в #0024).

**Drill-in:**

- Клик по кубику работающего агента → открывает в чат-табе ту сессию, в которой он сейчас работает (поведение #0026, переиспользуется).
- Клик по idle-кубику → открывает последнюю сессию этого агента в текущем ране (lastViewedSession persistence уже есть после #0026).
- Клик по User → открывает корневую user↔product сессию рана.

**Что выкидывается из #0023–#0026:**

- Стрелки коммуникации между кубиками как часть лэйаута (заменяются live-анимацией поверх — см. #0031, отдельная задача).
- `bridgeSessionId` как edge data в текущем graph-layout — переезжает в модель сессий (см. #0030, многоучастниковые комнаты).
- E2E TC-39 (drill через клик по edge) переписать или дропнуть: edge’ей в этом смысле больше нет.

**Что переиспользуется:**

- Drill-resolver (#0026) — owner-based матчинг сессии к кубику. Подходит без изменений.
- Live-state поллинг (#0024).
- `data-canvas-drill-session` контракт.

## Implementation notes

- Canvas-фича уже выделена в `src/webview/features/canvas/`. Лэйаут — отдельный модуль; заменить flow-layout на hierarchy-layout. Drill-resolver не трогать.
- Иерархия описывается константой (упорядоченный массив ролей + map role→level). Если в будущем появится 5-я роль, добавляется одна строка.
- Линии-«репортинги» рисуются один раз на маунт; не перерисовываются на каждый tick поллинга. Только статусы кубиков обновляются.
- Не закладываться на drag-and-drop — это не про эту задачу.
