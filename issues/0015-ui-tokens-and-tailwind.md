## Context

Текущий webview оформлен «программистски»: точечные inline-стили, прямые цвета, нет единой шкалы отступов. Чтобы дальнейшие задачи (компонентная библиотека #0016, полишинг чата #0020–#0022, canvas #0023+) могли строиться на общем фундаменте, нужен слой токенов.

Ключевая особенность: цвета **обязаны** идти из CSS-переменных VS Code, иначе переключение темы (light/dark/high-contrast) ломает внешний вид. Tailwind 4 (CSS-first) удобно ложится на эту модель — кастомные `@theme` ссылаются на `var(--vscode-…)`, и компоненты пользуются семантическими классами.

Это инфраструктурная задача без видимых пользователю изменений. Цель — чтобы после её закрытия любой следующий компонент использовал `bg-surface` / `text-muted` / `border-subtle` вместо ad-hoc стилей.

## Acceptance criteria

- В `package.json` webview'а добавлены `tailwindcss@^4` и `@tailwindcss/vite` (или эквивалент). Сборка webview (vite) обрабатывает Tailwind, итоговый CSS встраивается в bundle.
- Семантические токены определены через `@theme` (Tailwind 4 CSS-first), маппинг — на `var(--vscode-*)`. Минимальный набор:
  - **Поверхности**: `--color-surface`, `--color-surface-elevated`, `--color-surface-overlay`.
  - **Текст**: `--color-foreground`, `--color-muted`, `--color-disabled`.
  - **Бордеры**: `--color-border`, `--color-border-subtle`, `--color-border-focus`.
  - **Акценты**: `--color-accent`, `--color-accent-foreground`, `--color-danger`, `--color-warning`, `--color-success`.
  - **Роли агентов** (для канваса и аватаров): `--color-role-product`, `--color-role-architect`, `--color-role-user`, `--color-role-system` (изначально — производные от vscode badge/charts палитры).
- Шкала отступов и типографики — стандартная Tailwind (`space-1..12`), без кастомных значений на этом этапе.
- Смена темы VS Code применяется без перезагрузки webview (CSS-переменные живут на корневом элементе, Tailwind не «замораживает» цвета в build-time для этих токенов).
- В `RunDetails.tsx` (или одном небольшом компоненте) проведён демонстрационный рефакторинг 1–2 элементов на новые классы — как живая проверка работоспособности и пример для следующих PR.
- `lucide-react` добавлен в зависимости (используется в #0016+); базовый импорт-тест присутствует (например, заменена одна emoji-иконка в существующем UI).

## Implementation notes

- Tailwind 4 не требует `tailwind.config.js`; вся конфигурация — в одном CSS-файле (`src/webview/app.css` или подобном) через `@import "tailwindcss";` + `@theme { … }`. Это снижает порог входа и не плодит лишних артефактов.
- Vite-плагин `@tailwindcss/vite` подключается в [src/webview/vite.config.ts](src/webview/vite.config.ts) (если файла нет под таким именем — найти текущий конфиг webview).
- Для high-contrast темы VS Code некоторые переменные имеют другие значения; маппинг `bg-surface = var(--vscode-editor-background)` корректно покрывает все три темы без дополнительных условий.
- Не вводить параллельно никакой CSS-in-JS / CSS-модулей **в этой задаче**. CSS-модули остаются опцией для отдельных «нестандартных» компонентов в #0016+, но для базы — только Tailwind.
- ESLint/typecheck не должны сломаться: класснеймы — обычные строки, типизация не страдает. Если в проекте включён class-validator (типа `tailwind-merge`/`clsx`) — добавить только `clsx` (минимум, без `tailwind-merge` пока — нет конфликтов классов на этом этапе).
- TC отдельный не нужен: задача проверяется визуально + тем фактом, что демонстрационный рефакторинг в `RunDetails.tsx` собирается и работает в существующих e2e (TC-31/32 не должны упасть).
