## Context

Без инструментов любая роль (продакт, архитектор, программист) превращается в «функцию от prompt»: ничего не помнит между ранами, не может задать вопрос пользователю, не оставляет следов своей работы. Это убивает идею мульти-агентной системы — роли становятся тонкими прослойками поверх одной модели.

Поэтому tool runtime — фундамент, который нужен **до первой реальной роли**. Делаем его один раз, дальше все роли (#0003, #0004 и далее) живут поверх него.

OpenRouter поддерживает tool-calls в OpenAI-совместимом формате (поле `tools` в запросе, `tool_calls` в ответе, follow-up message с `role: "tool"`). Используем это напрямую — никакого собственного «протокола». Никаких frameworks (LangChain и т.п.) — `fetch` + цикл.

**Build vs buy.** Рассматривали Vercel AI SDK (единственный realistic-кандидат: LangChain слишком opinionated, Mastra молодая). Отказались по трём причинам: (1) `ask_user` должен переживать закрытие VS Code — generic-библиотеки рассчитаны на in-memory loop в одном процессе, обвязка поверх них съедает их же выгоду; (2) lock-in на их типы и темп breaking-изменений; (3) для OpenRouter-клиента уже выбран `fetch` без SDK — несогласованно тянуть SDK уровнем выше. Свой loop — ~80–120 строк, durability добавляет ещё ~50, и эти 50 пришлось бы писать поверх любой библиотеки. Пересмотреть решение, если появятся ≥3 ролей с расходящейся loop-логикой.

## Acceptance criteria

- В `src/extension/shared/openrouter/client.ts` (или соседнем модуле) появляется поддержка tool-calls: запрос принимает массив `tools` (JSON Schema каждого), ответ парсит `tool_calls`, есть хелпер для подачи `role: "tool"` обратно в следующий запрос.
- Появляется **tool-loop**: модуль `src/extension/shared/agent-loop/` (или аналогично), который принимает (system prompt, начальное user-message, набор зарегистрированных тулов, максимум итераций) и крутит цикл «модель → tool-call → выполнение → tool-result → модель» до тех пор, пока модель не вернёт finish_reason без tool-calls. Лимит итераций — константа (например, 20); при превышении — `failed` с понятной ошибкой.
- Каждый шаг цикла (assistant message, tool call, tool result) пишется в новый файл `.agents/runs/<id>/tools.jsonl` (append-only, по строке на запись). `chat.jsonl` остаётся для «человекочитаемой» ленты сообщений между ролью и пользователем — туда дублируются только финальные assistant-ответы и сообщения от/к пользователю, не каждый внутренний tool-call.
- Базовый набор тулов (зарегистрирован централизованно, доступен любой роли — конкретные роли решают, какие подмножества разрешать):
  - `kb.read({ path })` — читает файл из knowledge base. Путь относительный, sandbox только в `.agents/knowledge/`.
  - `kb.write({ path, content })` — атомарно (temp + rename) пишет файл в kb. Создаёт директории.
  - `kb.list({ path })` — листинг директории в kb.
  - `kb.grep({ pattern, path })` — `ripgrep`-стайл поиск по kb (можно через node-glob + чтение, ripgrep не тащим в зависимости).
  - `ask_user({ question, context })` — синхронный вопрос пользователю. Tool-loop останавливается, статус рана идёт в `awaiting_user_input`, в webview приходит событие с вопросом, появляется UI для ответа, ответ возвращается как tool-result, цикл продолжается. Если пользователь закроет VS Code — на следующем запуске ран остаётся в `awaiting_user_input`, ответ можно дать позже (по сути, синхронность только с точки зрения модели — для пользователя пауза неограниченная).
- Каждый tool валидирует входы по JSON Schema. Ошибка валидации — tool-result с `error: "..."`, модель видит ошибку и может попробовать снова (это ожидаемое поведение, не failure).
- Sandbox: `kb.*` тулы **не имеют доступа** за пределы `.agents/knowledge/` (никакого `..`, никаких абсолютных путей). Чтение/запись произвольных файлов проекта — отдельная будущая задача (нужна для программиста, не для продакта).
- Ошибки runtime (модель вернула невалидный tool_call, тул бросил исключение, лимит итераций) переводят ран в `failed` с записью в `tools.jsonl` и понятным сообщением в `chat.jsonl`.

## Implementation notes

- Tool registry — обычный `Map<string, ToolDefinition>` где `ToolDefinition = { name, description, schema, handler }`. Никакой динамики, никаких classes ради classes.
- Регистрация тулов — на старте каждой роли: роль сама собирает свой набор и передаёт в `runAgentLoop(...)`. `ask_user` доступен всем по умолчанию.
- `ask_user` — единственный «асимметричный» тул: его handler не выполняется синхронно внутри extension, а пробрасывается через IPC в webview и ждёт ответа. Реализация: новое IPC-событие `runs.askUser` (extension → webview) и `runs.userAnswer` (webview → extension); в extension это `Promise`, который резолвится по приходу ответа. Pending `ask_user` восстанавливается из `tools.jsonl` при повторном открытии VS Code (последняя запись = неотвеченный вопрос).
- Статусы рана пополняются: к `draft | running | awaiting_human | done | failed` добавляется `awaiting_user_input` (вопрос от агента) — отличается от `awaiting_human` (роль завершила свой шаг, ждёт approve).
- IPC: новые события `runs.askUser`, `runs.userAnswer`, `runs.toolEvent` (опционально — для live-апдейта tools-feed в UI; на первой итерации можно не показывать).
- TC писать **до реализации** — см. AGENT.md → раздел про тестовые артефакты. Минимум: успешный цикл с одним tool-call, цикл с `ask_user`, валидация сломанных входов, превышение лимита итераций, попытка `kb.read` за пределы sandbox.

## Outcome

Реализовано в две фазы:

- **Фаза A** — расширение OpenRouter-клиента под tool-calls, agent-loop с лимитом итераций, ajv-валидация входов, `tools.jsonl` лог, базовый набор `kb.read/write/list/grep` со sandbox в `.agents/knowledge/`, smoke-команда `AI Frontend Agent: Run Tool Loop Smoke`. TC-11 пройден ручным прогоном.
- **Фаза B** — тул `ask_user` поверх pending-asks registry + IPC-events `runs.askUser` / `runs.userAnswer`, новый статус `awaiting_user_input`, broadcast bus для уведомлений webview, форма ответа в `RunDetails`, durability через `loop.json` + `reconstructHistory` (resume после перезапуска VS Code). TC-15 и TC-16 пройдены ручным прогоном.

**Build vs buy** зафиксировано в Context — пишем свой loop, Vercel AI SDK отброшен из-за durability-требований.

**Не сделано (edge-кейсы — оставлены на естественное обнаружение в #0002–#0004):**

- TC-12 (sandbox bypass) и TC-13 (невалидные входы) формально не прогнаны, но код реализован — `resolveKnowledgePath` и `validateToolArgs` покрывают эти кейсы.
- TC-14 (лимит итераций) — искусственная провокация малополезна, проверим если реально упрёмся.
- Автоматический ребродкаст pending-вопроса при подключении нового webview не делал — пользователь должен сам кликнуть на ран. На текущем UX приемлемо.

**Известные ограничения, которые могут стать задачами:**

- gemini-3.1-flash-lite-preview не всегда надёжно вызывает `ask_user` — пришлось ужесточать system prompt smoke-роли. Для реальных ролей (#0003 продакт, #0004 архитектор) скорее всего понадобится более умная модель.
- Нет UI для просмотра `tools.jsonl` — диагностика только через файлы. Если станет узким местом — добавим вкладку «tools log» в `RunDetails`.
