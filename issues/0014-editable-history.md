## Context

Сейчас сообщения в `chat.jsonl` append-only без возможности правки. Разговор с агентом — итеративный процесс уточнений; невозможность вернуться и поправить превращает мелкую ошибку (опечатка в требовании, неточная формулировка) в полное переписывание рана. Сюда же — старые (компактнутые после #0013) сессии: они read-only, и это временное упрощение.

Эта задача отделена от #0012 (мульти-агент), поскольку storage-формат `chat.jsonl` и UI-кнопки Edit/Revert ортогональны agent-agent сессиям и могут идти параллельно.

## Acceptance criteria

- В UI у каждого сообщения (любого `from`) появляется «Edit» (иконка/контекстное меню). Доступно для `user`, `agent:product`, `agent:architect`, `agent:system` сообщений.
- Редактирование пишет новую запись `{type: 'edit', id, replaces, text, at}` в `chat.jsonl`. Оригинал остаётся в файле, в effective-state помечается `replacedBy: newId`.
- В ленте по умолчанию показывается актуальная версия. У отредактированных сообщений есть бейдж «edited» и раскрывающаяся «история правок» (показывает все версии в порядке времени).
- Откат: «Вернуть как было» — добавляет ещё одну edit-запись, которая воскрешает оригинальный текст (никогда не удаляем строки из jsonl).
- При резюме / новой итерации agent-loop'а в model context подаётся **актуальная** версия. `reconstructHistory` сворачивает edit-цепочки заранее.
- Edit пользовательского сообщения: если сообщение находится в середине истории (после него были assistant-шаги), правка инвалидирует все последующие сообщения этой сессии. UI показывает confirm: «Сообщения после этой точки будут скрыты в новой ветке. Продолжить?». Скрытие — через snapshot `forkedAt: messageId` в SessionMeta + флаг `hidden: true` для последующих записей.
- Edit assistant-сообщения: только косметическая правка (опечатки, форматирование). Не запускает re-execution loop'а.
- Снапшоты диска версионирует git, который и так трекает `.agents/` — отдельный backup-механизм не вводим.
- Старые сессии после #0013 (`status: 'compacted'`) — composer остаётся disabled, но Edit на сообщениях работает (для исправления записей задним числом). Это явное отступление от текущего «compacted = read-only».

## Implementation notes

- Storage: `chat.jsonl` уже append-only; новые записи укладываются без миграций.
- Read-функция (`readChat(sessionId)`) собирает «эффективное» состояние, проходя по replaces-цепочке: для каждого `id` берёт последнюю edit-запись.
- Reconstruct в agent-loop: `reconstructHistory` фильтрует `hidden: true` и применяет edits ДО конвертации в OpenRouter messages.
- IPC: новый `runs.message.edit { runId, sessionId, messageId, text }`, broadcast `runs.message.edited`. Откат — тот же handler с оригинальным текстом, без отдельного endpoint'а.
- UI: `EditMessageModal` — textarea + Save/Cancel; диалог инвалидации — отдельная `ConfirmModal` (если её ещё нет — взять из #0013).
- TC: правка пользовательского сообщения в середине истории скрывает последующие, актуальный текст уходит в loop при resume; правка assistant-сообщения не триггерит loop; история правок видна через UI.
