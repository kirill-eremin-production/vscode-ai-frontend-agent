# Issues

Текстовый трекер задач. Каждая задача — отдельный `.md` файл с порядковым номером в имени: `NNNN-короткое-название.md` (`0001-architect-role.md`, `0002-tool-runtime.md`).

## Зачем

Внешний tracker (GitHub Issues, Linear) добавил бы лишнюю зависимость на этапе, когда разработчик и пишущий код агент — одно и то же лицо. Файлы в репо живут рядом с кодом, попадают в git вместе с ним, читаются и правятся теми же инструментами.

## Жизненный цикл

Статус хранится в frontmatter (`status: open | in-progress | done | dropped`). Закрытые задачи **не удаляем** — они нужны как контекст «почему мы вообще это сделали» при будущих доработках.

При закрытии задачи (`done` или `dropped`) **переименовываем файл**: добавляем префикс `DONE-` к имени, например `0001-tool-runtime.md` → `DONE-0001-tool-runtime.md`. Это даёт мгновенный визуальный сигнал в листинге `issues/` без открытия файлов и поднимает активные задачи наверх при сортировке.

Если задача отменена без реализации — `status: dropped` + одна-две строки в секции `## Outcome` с причиной. Перенумеровывать ничего не нужно: пропуски в нумерации допустимы.

## Структура файла

```markdown
---
id: NNNN
title: <короткое описание>
status: open
created: YYYY-MM-DD
---

## Context

Что и почему делаем. Достаточно, чтобы холодный читатель (или агент) понял задачу без чтения чата.

## Acceptance criteria

- Чек-лист конкретных проверяемых результатов.
- Каждый пункт должен мапиться либо на user-story, либо на новый test-case.

## Implementation notes

Опционально: подсказки про модули, файлы, тонкости. Не план шаг-в-шаг — только то, что не выводится из кода и не должно быть забыто.

## Related

- US-N, TC-N — что подтверждает выполнение.
- #NNNN — связанные задачи.
```

## Правила

- Нумерация сквозная и неубывающая. Удалённые номера не переиспользуем.
- При закрытии задачи добавляем секцию `## Outcome` с кратким описанием реализованного и ссылками на коммиты/PR (если применимо).
- Когда меняется приоритет/требования — правим тот же файл, а не плодим новый. История остаётся в git.
