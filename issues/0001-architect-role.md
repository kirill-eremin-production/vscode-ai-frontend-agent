## Context

Сейчас при `Start run` мы только сохраняем prompt пользователя и красивый заголовок — никакая роль ещё не запускается. Чтобы валидировать всю агентскую цепочку end-to-end на минимально полезном кейсе, нужна одна реальная роль. Выбран **архитектор**: его выход (`plan.md`) дешёвый, текстовый, не трогает рабочий код проекта и сразу даёт пользователю видимый артефакт. Остальные роли (программисты, ревьювер) логически зависят от плана архитектора, поэтому он же — естественный первый шаг.

Tool runtime (read/write/grep/ask) на этой задаче **не делаем**. Архитектор пока работает «вслепую»: получает на вход только prompt пользователя, не читает файлы проекта. Это сознательное ограничение первой итерации — закрываем шаг 4 из плана (см. историю чата `roles → orchestrator`), tool runtime откладываем на отдельную задачу.

## Acceptance criteria

- При создании рана сразу после `chat.jsonl` стартует архитектор.
- Используется отдельная (более «умная») модель, имя константой в `src/extension/entities/run/roles/architect.ts` (или аналогичном месте). Модель захардкожена, без settings.
- Архитектор получает system prompt, описывающий его роль и формат вывода (структурированный markdown: цели → разбиение на подзадачи → чек-лист рисков).
- Ответ модели сохраняется в `.agents/runs/<id>/plan.md` атомарно (temp + rename), как сейчас пишется `meta.json`.
- В `chat.jsonl` появляется сообщение с `from: "agent:architect"` и текстом плана (или ссылкой/превью — решить в процессе).
- Статус рана меняется: `draft → running` на старте архитектора, `running → awaiting_human` после успеха (план готов, ждём approve пользователя). При сбое — `failed`, причина в новом `decisions.md` или в самом сообщении.
- В UI карточка рана показывает план (можно через тот же `RunDetails` — добавить секцию). Кнопок approve/reject пока нет: «awaiting_human» отрисовывается просто статусом.
- Если ключ OpenRouter невалиден или сеть упала — ран помечается `failed`, пользователь видит понятное сообщение, на диске остаётся диагностика.

## Implementation notes

- Перед стартом роли service-слой должен подгружать ключ через `getOpenRouterKey` (как уже делается для title).
- Запуск архитектора — **fire-and-forget** из `createRun`: не блокируем IPC-ответ, шлём `runs.created` сразу, далее по мере прогресса шлём в webview обновления (новое сообщение в IPC: `runs.updated` с целым `RunMeta`, или `runs.message.appended`).
- В webview store нужен обработчик нового события и обновление `selectedDetails`, если открыт текущий ран.
- Ошибки роли и обновления статуса централизуем в `service.ts` — добавить функцию `runArchitect(runId)` и `updateRunStatus(runId, status)` поверх `writeMeta`.
- Контракт IPC живёт в `src/extension/features/run-management/messages.ts` — продумать имя нового события и **добавить тест-кейс в `tests/manual-test-cases.md` ДО реализации** (см. AGENT.md → раздел про тестовые артефакты).
