## Context

Сейчас UI неотзывчив: пока агент думает или вызывает тулы, в ленте долго ничего не появляется, в шапке нет никакого индикатора, в кнопках после клика — пустота. Пользователь не понимает, жив ли процесс.

Нужен единый слой «осмысленных лоадеров»: один источник истины — функция, которая по `meta.status` + последнему активному tool-event отдаёт человеческую строку, и потребители этого источника в шапке рана, на кнопках, на канвасе (#0024).

## Acceptance criteria

- Новая утилита `src/webview/shared/lib/run-status-caption.ts` — чистая функция:
  ```ts
  function describeRunActivity(input: {
    meta: RunMeta;
    tools: ToolEvent[]; // tools видимой сессии, в хронологическом порядке
    role: AgentRole; // роль, чью активность описываем (для канваса; для шапки — активная роль)
  }): {
    kind: 'idle' | 'thinking' | 'tool' | 'awaiting_user' | 'awaiting_human' | 'failed';
    label: string;
  };
  ```
  Маппинг строк (как в US-23):
  - `draft` → `idle`, «Готовлюсь к запуску…»
  - `running`, последний tool — pending → `tool`, «{Роль}: вызов `{tool}`…»
  - `running`, активного tool нет → `thinking`, «{Роль} думает…»
  - `awaiting_user_input` → `awaiting_user`, «{Роль} ждёт твой ответ»
  - `awaiting_human` → `awaiting_human`, «Готово — твой ход»
  - `failed` → `failed`, «Ошибка: {краткое описание из meta или последнего tool.error}»
- Шапка main-area (`RunHeader.tsx` или подобный): рядом с заголовком рана отображается активность через атом `LoadingState` (#0016), цвет по `kind` (idle — приглушённый, thinking/tool — accent, awaiting_user — warning, failed — danger).
- Любая кнопка, инициирующая IPC (создать ран #0018, отправить сообщение, переключить сессию #0019, ответить на ask_user, кнопки канваса #0026) — на время `pending` показывает inline-спиннер с подписью действия и блокируется. Это — поведение `Button loading={...} loadingLabel="..."` атома из #0016.
- Skeleton'ы (#0016) подключаются:
  - В левой панели на момент загрузки списка ранов.
  - В main-area на момент загрузки деталей выбранного рана (до того, как пришёл первый `runs.get`-ответ).
- «Долго отвечает»-подсказка: если конкретный pending IPC висит дольше порога (`runs.create` 10s, `runs.send` 5s, `runs.get` 5s), рядом со спиннером появляется текст «Долго отвечает — проверь сеть или ключ» и кнопка «Отменить». Отмена выставляет `pending: false` локально (IPC-запрос уходит «в воздух», но UI больше не блокирует пользователя).
- Если у пользователя `prefers-reduced-motion: reduce` — анимации спиннеров заменяются на статичную точку, lit/dim'ом подписывают «работаю»/«жду».

## Implementation notes

- `describeRunActivity` — чистая функция, легко юнит-тестируется. Покрыть всеми ветками маппинга в новом `*.spec.ts` рядом.
- «Активная роль» для шапки рана выводится из активной сессии: в user-agent сессии — роль агента из `participants`; в agent-agent (bridge) — роль _приёмника_ handoff'а (та, которая сейчас в фокусе цикла) — определяется по последнему сообщению в bridge (если последний — от продакта-источника, то приёмник архитектор; иначе — последний автор).
- Pending-state IPC-запросов — вынести в маленький helper в store (`startPending(key)` / `endPending(key)`) и продолжать жить там же, где сейчас живут IPC-обёртки. Кнопки получают своё `pending` через селектор по ключу.
- Длинные таймауты — простой `setTimeout` в обёртке IPC; не переусложнять.
- TC: юнит-тесты для `describeRunActivity` (минимум 6 кейсов — по одному на каждый `kind`); опциональный e2e — после старта рана в шапке появляется строка «Продакт думает…», после ответа brief — «Готово — твой ход».
