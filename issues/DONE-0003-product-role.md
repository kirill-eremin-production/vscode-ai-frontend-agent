## Context

Первая «настоящая» роль в системе. Продакт превращает сырой prompt пользователя в структурированный продуктовый бриф (`brief.md`). В отличие от stub-варианта, у продакта есть инструменты — он реально работает как продуктовый менеджер:

- читает свою базу знаний (`#0002`), чтобы вспомнить, что уже обсуждалось по этой фиче и каких решений придерживались;
- задаёт пользователю уточняющие вопросы через `ask_user` (`#0001`), а не выдумывает acceptance criteria;
- пишет в kb новые decisions, обновляет описания фич, фиксирует открытые вопросы — следующий ран по той же области стартует не с нуля.

Без этого продакт = «функция от prompt», см. обсуждение в чате при заведении задач.

Зависит от **#0001** (tool runtime, `ask_user`, `kb.*` тулы) и **#0002** (структура kb продакта). Без них не имеет смысла.

## Acceptance criteria

- При создании рана сразу после `chat.jsonl` стартует продакт через `runAgentLoop` из #0001.
- Модель — отдельная константа в `src/extension/entities/run/roles/product.ts`. Для продакта не нужна топовая модель (он не пишет код), но достаточно умная для текстового анализа и работы с тулами. Конкретное имя выбираем при реализации, фиксируем литералом.
- Продакт получает system prompt, который описывает:
  - его роль (продуктовый менеджер фронтенд-проекта);
  - доступные тулы и **обязательный паттерн работы**: сначала `kb.list` + `kb.grep`/`kb.read` по релевантным разделам — потом анализ — потом `ask_user` для пробелов — потом `kb.write` для новых артефактов — потом финальный `brief.md`;
  - формат `brief.md` (фиксированный — см. ниже);
  - правила kb (полностью копируем содержимое `.agents/knowledge/product/README.md` или ссылаемся на него — финализируем при реализации).
- Доступные продакту тулы: `kb.read`, `kb.write`, `kb.list`, `kb.grep` (sandbox `.agents/knowledge/product/`), `ask_user`. Другие тулы (включая чтение файлов проекта) — **не выдаём**.
- Финальный `brief.md` сохраняется в `.agents/runs/<id>/brief.md` атомарно. Жёсткий формат:
  - `# <название задачи>`
  - `## Проблема`
  - `## Целевой пользователь и сценарий`
  - `## User stories`
  - `## Acceptance criteria`
  - `## Не-цели`
  - `## Связанные артефакты kb` — список путей в kb, на которые продакт опирался или которые создал/обновил в этом ране.
- Секция `## Открытые вопросы` **отсутствует** — у продакта есть `ask_user`, нерешённые вопросы он либо задаёт, либо явно фиксирует в kb (`questions/`) с пометкой «отложено». Это и есть детерминированность: либо ответ есть, либо есть явное место, где он отложен.
- В `chat.jsonl` появляется сообщение с `from: "agent:product"` и текстом брифа (или превью). Каждый `ask_user` тоже отражается в `chat.jsonl` как пара сообщений `from: "agent:product"` (вопрос) и `from: "user"` (ответ) — для человекочитаемой ленты. Полный лог тулов остаётся в `tools.jsonl`.
- Статусы: `draft → running → awaiting_user_input` (на каждом `ask_user`) → `running` (после ответа) → `awaiting_human` (бриф готов, ждёт approve) | `failed`.
- В UI карточка рана показывает бриф (если готов) и текущий вопрос с полем ответа (если `awaiting_user_input`). Approve/reject кнопок пока нет — `awaiting_human` отрисовывается просто статусом.
- Если ключ невалиден или модель упала на любом шаге — ран `failed`, диагностика в `tools.jsonl` и сообщение в `chat.jsonl`.

- Формат `brief.md` — контракт для архитектора (#0004). Зафиксировать сейчас, чтобы потом не переписывать парсер вместе с prompt-ом.
