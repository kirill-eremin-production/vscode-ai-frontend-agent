## Context

Иерархия на canvas (#0028) показывает «кто есть в команде и кто чем занят прямо сейчас», но не отвечает на вопрос «что произошло за этот ран и в каком порядке». Эту роль берёт на себя хронологический список встреч — по сути, лента чатов рана, упорядоченная по времени старта.

Метафора: журнал встреч в реальной команде. Каждая запись — это «встреча» (= сессия чата) с указанием участников, инициирующего входа и того, что родилось из этой встречи.

## Acceptance criteria

**Модель записи о встрече:**

Каждая запись — это сессия рана, обогащённая полями:

- `participants: Role[]` — кто реально вёл диалог. **Не считаем участником** того, чьим артефактом сессия только инициирована (например, продакт передал brief архитектору — продакт **не** участник этой сессии, он только источник input’а).
- `inputFrom: Role | 'user'` — кто/что инициировало (либо роль автора артефакта-входа, либо пользователь напрямую).
- `prev: SessionId[]` — сессии, из которых эта родилась (чаще всего одна; массив — для будущих merge-кейсов).
- `next: SessionId[]` — сессии, рождённые из этой (массив — потому что одна встреча может породить несколько последующих: например, архитектор после встречи с продактом инициирует и работу программиста, и записи в kb-сессию).
- Тайминги: `startedAt`, `endedAt | null`.
- Статус: `active` | `finished` | `paused` (см. #0031).

**UI-панель:**

- Новая панель «Встречи» в side-area или вкладке (рядом с sessions panel или вместо неё — финализировать при дизайне). Лента сверху-вниз по `startedAt` desc (свежие сверху), либо asc — выбрать при реализации, важно консистентно.
- Каждая карточка встречи показывает: участников (аватары/иконки ролей), `inputFrom`, время старта, статус, превью первого/последнего сообщения.
- Клик по карточке → открывает чат-таб на этой сессии (как drill-in в #0026).
- prev/next отображаются как ссылки «← откуда» / «→ что родилось» внутри карточки. Клик навигирует к соответствующей карточке/сессии.

**Поведение по сценариям из обсуждения:**

1. User → Product: создаётся встреча `{participants: [product], inputFrom: 'user', prev: [], next: []}`. По мере handoff’ов её `next` пополняется.
2. Product → Architect (handoff с brief.md): новая встреча `{participants: [architect], inputFrom: 'product', prev: [user-product-session], next: []}`. У user-product-session в `next` добавляется ID этой новой.
3. Architect → Programmer (handoff с plan.md): аналогично, `participants: [programmer], inputFrom: 'architect'`.
4. Programmer → Product (вопрос «снизу» через уровень): новая встреча `{participants: [programmer, architect, product], inputFrom: 'programmer', prev: [architect-programmer-session], next: []}` — обязательно с архитектором (см. #0032).
5. Architect → Product: `{participants: [architect, product], inputFrom: 'architect', prev: [...], next: []}`. Архитектор может также пригласить программиста, если хочет, чтобы тот сразу получил ответ — см. #0030 (pull-into-room).

**Что не делаем в этой задаче:**

- Не делаем поиск/фильтр по ленте (отложено).
- Не делаем экспорт ленты.
- Per-message timeline внутри встречи — это уже текущий чат-вью, не дублируем.

## Implementation notes

- Модель уже близка: сессии с `parentSessionId` есть из мульти-агента. `prev` ≈ `parentSessionId` в массивной форме; `next` — производная (можно вычислять обратным индексом, либо хранить явно для O(1) UI). Финализировать при реализации.
- `participants` — **не** список всех, кто хоть раз появился в логе сессии. Это поле определяется при создании сессии и при явных pull-into-room действиях (#0030). Не пересчитывается из messages. Иначе теряется различие «продакт инициировал работу архитектора» vs «продакт реально присутствовал в чате».
- `inputFrom` — для bridge-сессий это роль агента, чей артефакт ушёл во входное сообщение (product для brief, architect для plan). Для user-сессий — `'user'`.
- Storage: расширить session metadata (там, где сейчас живёт `parentSessionId`). Не вводить отдельную сущность Meeting — это и есть сессия с дополнительными полями.
