## Context

Сейчас все agent↔agent сессии — строго парные (bridge между двумя ролями). Новая модель команды (#0029) требует сессий с N>2 участниками: программист задаёт вопрос продакту с архитектором в комнате; архитектор зовёт продакта и программиста, чтобы тот сразу получил ответ.

Это первое архитектурное расширение, которое ломает имплицитное предположение «у session.participants всегда длина 2».

## Acceptance criteria

**Модель:**

- `Session.participants: Role[]` — массив, не пара. Длина ≥ 1 (для user-only edge-кейсов) и ≥ 2 на практике для agent-сессий.
- Сообщение в сессии хранит `authorRole: Role` (как сейчас, не меняется). Подписка на «свои сообщения» агента — фильтр по `authorRole`.
- Новый action: `pullIntoRoom(sessionId, role)` — добавляет роль в `participants` существующей активной сессии. С этого момента приглашённый агент видит всю историю сессии и может писать в неё.
- В лог сессии при `pullIntoRoom` пишется системное событие `participant_joined` с `role` и `at` — чтобы UI и резумер могли отделить «то, что было до прихода» от «то, что после».

**Поведение агента в комнате:**

- Системный промпт агента, попавшего в сессию через pull-in, начинается с короткого префикса «ты приглашён в активный чат, история выше — контекст; отвечай на последний вопрос». Точная формулировка — при реализации.
- Очерёдность: на первой итерации — простая. После каждого user/agent-сообщения система решает, кто из участников должен ответить (round-robin? последний адресуемый? — финализировать при реализации, не закладываться на сложную модель). Допустимо: «отвечает тот, к кому обратились по имени роли в последнем сообщении; если ни к кому — отвечает инициатор».
- Не делаем параллельных одновременных ответов — в комнате говорит один агент за раз, остальные ждут.

**UI:**

- В чат-вью видно `participants` сессии (список аватаров наверху). При `pullIntoRoom` UI обновляется live.
- В карточке встречи (#0029) `participants` отражает текущий состав, не исходный.
- Сообщения в чате визуально различаются по `authorRole` (роль-цвет/аватар у каждого bubble) — на тот случай, когда в одной сессии говорят 3+ роли.

**Storage migration:**

- Сессии, созданные до этой задачи, имеют `participants` длины 2 (восстанавливается из старых полей `agentRole`/тип сессии). Никакой миграции файлов на диске не требуется на чтении (read-time normalization). При write новых сессий — новый формат сразу.

**Тесты:**

- Unit на `pullIntoRoom`: добавление роли в активную сессию пишет `participant_joined`, обновляет `participants`, повторный вызов идемпотентен (если роль уже в комнате — no-op).
- Unit на read-time normalization старых session-файлов.
- E2E TC: программист зовёт продакта (с архитектором) → в чате три аватара, есть событие `participant_joined` для product и architect, продакт отвечает, архитектор может вмешаться.

## Implementation notes

- Где это ломает текущий код: любые места, делающие `if (session.type === 'agent-agent') { const [a, b] = session.participants }`. Это деструктурирование надо ревизовать. Грубый поиск — `participants[0]` / `participants[1]` / `.length === 2`.
- `pullIntoRoom` инициируется либо тулом агента (`team.invite(role)` — простейший вариант), либо UI-кнопкой пользователя «добавить в чат». Минимум на этой итерации — тул агента; UI-кнопка — будущее.
- Резумер: при подъёме сессии после рестарта VS Code участник, добавленный через `participant_joined`, должен корректно подцепиться (посчитать его историю с момента события, не с начала сессии).
