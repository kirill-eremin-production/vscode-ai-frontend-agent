## Context

Сейчас webview — двухколоночный: слева список ранов с composer'ом внизу, справа детали рана. По мере появления дерева сессий (#0012) и канваса (#0023) такой лэйаут перестаёт справляться: дерево сейчас живёт прямо в `RunDetails.tsx`, занимает место, путает чат и навигацию по сессиям; форма создания рана сидит в сайдбаре и конкурирует за внимание со списком.

Задача — переложить webview в трёхпанельную схему `[RunList] [MainArea] [SessionsPanel]` с обеими боковыми панелями collapsible, и убрать composer создания рана из левой панели. Создание рана уезжает в main-area отдельным экраном (#0018), дерево сессий — в правую панель (#0019).

## Acceptance criteria

- Корневой layout webview — три колонки. Левая (`RunListPanel`) и правая (`SessionsPanel`) — collapsible независимо. Состояние раскрытости каждой панели хранится в `globalState` extension'а и переживает перезапуск VS Code.
- `RunListPanel`:
  - Шапка с заголовком «Раны», кнопкой «+ Новый ран» (IconButton), кнопкой «Свернуть» (IconButton).
  - Тело — список ранов (как сейчас, US-3): заголовок, дата, статус, плюс компактный индикатор активной роли в `running`-ранах («работает: продакт»).
  - **Никакого composer'а внутри панели.** Поле ввода для нового рана здесь больше не появляется.
- В свёрнутом состоянии каждая панель остаётся видимой узкой полосой (~32px) с двумя иконками: «развернуть» и (для левой) «+ Новый ран». Клик по «развернуть» возвращает панель.
- Кнопка «+ Новый ран» в шапке списка диспатчит действие, которое переводит main-area в режим создания рана (контракт с #0018: store.action `startNewRun()` — пока обработчик в main-area может быть заглушкой, если #0018 ещё не сделан).
- Кнопка «Свернуть» правой панели по умолчанию — в шапке самой правой панели; реализация самой панели — заглушка (`<div>Sessions panel — TBD #0019</div>`), чтобы layout был валидным до прихода #0019.
- На узком окне (< ~700px) обе боковые панели по умолчанию свёрнуты; main-area занимает всю ширину. При расширении окна состояние не «вспоминается» автоматически — пользователь сам разворачивает.
- Существующая функциональность не регрессирует: выбор рана из списка по-прежнему открывает его детали в main-area (на этом этапе main-area продолжает рендерить текущий `RunDetails.tsx` с временной отметкой «дерево сессий переедет в правую панель в #0019»).
- Все atomic-кнопки и иконки — из библиотеки #0016 (`IconButton`, `Panel`).

## Implementation notes

- Новый компонент `src/webview/app/shell/AppShell.tsx` — отвечает только за layout (CSS Grid: `grid-template-columns: auto 1fr auto`, ширины боковых через CSS-переменные с двумя значениями: collapsed/expanded).
- Состояние collapsed: добавить два поля в webview store (`leftPanelCollapsed`, `rightPanelCollapsed`), persist'ить через IPC `state.setUiPref { key, value }` → extension сохраняет в `context.globalState`. На старте webview `state.getUiPrefs` возвращает мапу.
- Если IPC для UI-префов раньше не было — это первый кейс; сделать тонкий generic (любой ключ), чтобы #0024 (per-run last-viewed tab) и #0026 (canvas zoom) могли использовать тот же канал.
- Действие `startNewRun()` в store: ставит флаг `mainAreaMode: 'new-run'`. Главный компонент main-area (`MainArea.tsx`) переключается между `'new-run' | 'run-details' | 'empty'` на основе этого флага и `selectedRunId`. На этом этапе режим `'new-run'` рендерит заглушку (`<div>Форма создания — #0018</div>`) — реальная реализация в #0018.
- TC дополнить: TC-32/TC-31 не должны упасть. Добавить мини-проверку (можно отдельным TC или расширением существующего): при загрузке webview обе панели в дефолтном (раскрытом) состоянии видимы, кнопка «+ Новый ран» — в шапке списка, не в его теле.
