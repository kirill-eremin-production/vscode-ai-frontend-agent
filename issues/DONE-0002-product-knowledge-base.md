## Context

Продакт без памяти между ранами — это та самая «тупая прослойка», от которой мы отказались. Чтобы он мог формулировать осмысленные продуктовые требования, помнить решения, ссылаться на прошлые обсуждения и спорить с пользователем по делу — нужна персистентная база знаний.

KB живёт в **рабочем workspace пользователя**, рядом с кодом и `.agents/runs/`, и коммитится в git вместе с проектом — продуктовая память это часть проекта, а не личное хранилище агента. Это та же логика, что для `.agents/runs/`: документация и решения агентов живут рядом с кодом.

Эта задача — про **место и конвенцию**, не про роль. Сама роль продакта (#0003) использует kb через тулы из #0001. Без формализованной структуры продакт начнёт раскладывать файлы как попало — потом неудобно ни ему искать, ни человеку читать.

## Acceptance criteria

- Корневая директория: `.agents/knowledge/product/` в корне workspace. Создаётся лениво при первой записи через `kb.write`. В git коммитится (никакого `.gitignore` на неё).
- Поддиректории — фиксированная схема (продакт обязан класть файлы только сюда):
  - `glossary/` — определения продуктовых терминов, по файлу на термин (`<term-slug>.md`).
  - `personas/` — целевые пользователи (`<persona-slug>.md`).
  - `decisions/` — продуктовые решения с датой (`YYYY-MM-DD-<slug>.md`). Принцип ADR: контекст / решение / последствия.
  - `features/` — описания фич/областей продукта (`<feature-slug>.md`). Сюда же продакт складывает обобщённый опыт прошлых ранов по этой фиче.
  - `questions/` — открытые вопросы, которые продакт пока не задавал пользователю или не получил окончательный ответ (`<question-slug>.md`). Когда вопрос закрыт — файл переносится в `decisions/` или `features/`, не удаляется.
- Каждый файл — markdown с фронтматтером:
  ```yaml
  ---
  type: glossary | persona | decision | feature | question
  created: YYYY-MM-DD
  updated: YYYY-MM-DD
  related: [список путей внутри kb]
  ---
  ```
- Конвенция для `kb.grep` (#0001): продакт ищет по содержимому markdown — фронтматтер участвует в поиске наравне с телом, это даёт фильтрацию по `type:` бесплатно.
- В `.agents/knowledge/product/README.md` лежит описание схемы (что куда класть) — этот же файл agent читает в начале каждого рана, чтобы помнить правила. Это самодостаточный «system-prompt extension», файл генерируется один раз вручную (или скриптом инициализации) и дальше живёт в репо.
- Никакой автогенерации индекса. `kb.list` + `kb.grep` достаточно — индексный файл быстро устареет и будет врать.

## Implementation notes

- Эта задача почти не требует кода — основное это **написать `README.md` с описанием схемы** и убедиться, что `kb.*` тулы из #0001 работают с этими путями. Один helper в `src/extension/entities/knowledge/` для резолва пути `<role>/<subdir>/<file>` относительно workspace + проверка sandbox.
- Не делаем общий `knowledge/` для всех ролей сразу. Каждая роль получит свою (`knowledge/architect/`, `knowledge/programmer/`...) в своих задачах — структура у них будет разная (архитектор — про модули и решения, программист — про конвенции кода и т.п.). Шарить kb между ролями на старте — преждевременная унификация.
- Cross-role чтение (например, архитектор хочет посмотреть `knowledge/product/`) пока **не разрешаем**. Если когда-нибудь понадобится — обсудим отдельно. Изоляция упрощает sandbox.
- При запуске VS Code не делаем никакой миграции / валидации kb. Если файлы поломаны или схема съехала — продакт сам разберётся при чтении (или попросит пользователя через `ask_user`).
