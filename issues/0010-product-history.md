## Context

US-15 и US-16 — про эволюцию продактовой kb из «склад заметок» в живую память продукта: универсальность по доменам (не только разработка фич, но и маркетинг, дизайн, цели, pricing) + строгая история решений с supersede-цепочками.

Без этого через 5–10 ранов продактовая kb превращается в неинтерпретируемую кашу: ничто не помечено устаревшим, ничто не привязано к причине смены курса, новый ран не знает, на что опираться, и противоречия копятся молча.

Обе истории затрагивают одно и то же — kb-схему и формат финального артефакта — поэтому идут одной задачей.

## Acceptance criteria

US-15:

- Структура `.agents/knowledge/product/` расширена разделами под не-фичевые домены: минимум `marketing/`, `design-direction/`, `goals/` (или `okrs/`), `metrics/`, `pricing/`. Точный список финализируется при реализации; `PRODUCT_KB_README_MARKDOWN` (#0002) описывает схему как явно открытую («новые домены добавлять сюда же, по существующему шаблону»).
- Имя финального артефакта рана зависит от типа: `brief.md` для фич, `proposal.md`/`decision.md`/`plan.md` для не-фичевых обсуждений. Список вариантов — фиксированный enum в коде и в system prompt продакта (не свободный выбор).
- В [RunDetails.tsx](src/webview/features/run-list/ui/RunDetails.tsx) карточка показывает имя/тип финального артефакта (а не предполагает, что любой ран обязан произвести `brief.md`).
- System prompt описывает продакта как «директора продукта» с ответственностью за весь жизненный цикл, не как «писателя ТЗ для разработки».

US-16:

- System prompt содержит обязательный паттерн: на старте рана — `kb.list` + `kb.grep` по релевантным разделам, явное сравнение с прежними решениями, при конфликте — supersede-цепочка.
- Frontmatter convention для decisions: `status: active | superseded`, `superseded_by: <относительный путь>`, `superseded_at: YYYY-MM-DD`, `supersedes: [<пути>]`. Описано в README продакта.
- При смене решения продакт обновляет frontmatter старых артефактов (через `kb.write`) — не удаляя их.
- Каждое решение содержит обоснование («почему», рассмотренные альтернативы, что отвергли и по какой причине). Заглушки вида «решено: X» без обоснования — нарушение, фиксируется в system prompt как недопустимое (поведенческая проверка через TC).
- Каждая фича/раздел kb имеет в начале файла актуальное описание + явный список ссылок на сформировавшие его decisions; при смене решения описание обновляется.
- Финальный артефакт рана в секции `## Связанные артефакты kb` перечисляет и прочитанные, и созданные/обновлённые/устаревшие пути.
- При обнаружении «висящих» противоречий (несколько активных decision-ов друг против друга по одной теме) первым шагом — `ask_user` с просьбой подтвердить актуальность.

## Implementation notes

- Расширение `PRODUCT_KB_README_MARKDOWN` (#0002) — туда же описание новых разделов и frontmatter convention для decisions. README остаётся единственным источником правды по схеме продактовой kb (пользователь видит его в репозитории, продакт получает целиком в system prompt).
- Имя финального артефакта: ввести в `runProduct` маленький enum + writer per type (`writeBrief` / `writeProposal` / `writeDecision` / …). Все писатели — через [storage.ts](src/extension/entities/run/storage.ts) (AGENT.md «Структуру правит только storage.ts»). Тип артефакта продакт выбирает сам по типу запроса; код просто маршрутизирует выбор в правильный writer.
- Sandbox kb продакта (`buildRoleScopedKbTools('product')` в [role-kb-tools.ts](src/extension/features/product-role/role-kb-tools.ts)) уже на месте — расширение разделов работает «само», дополнительно префиксовать ничего не нужно.
- Supersede-механика — целиком на уровне system prompt + frontmatter convention. Код её не валидирует (нет YAML-парсера для frontmatter в kb-тулах сейчас, и заводить только ради этого валидатора не стоит). Контроль — через TC и через ревью kb-папки человеком.
- TC писать **до реализации**: TC-32 (маркетинговый ран → `proposal.md`, разделы `marketing/`/`goals/` наполнены), TC-33 (supersede-цепочка из обсуждения: средневековье → космос; старый decision и feature-файлы помечены `superseded` со ссылкой на новый), TC-34 (ран при наличии «висящих» противоречий первым шагом задаёт `ask_user`). Номер TC-31 занят #0004 (architect happy path).
- US-15 и US-16 — два больших куска. Если при реализации станет тесно — US-15 можно вынести отдельно; пока держим вместе, потому что обе ставят правила kb-схемы и легче не разъезжаться.
