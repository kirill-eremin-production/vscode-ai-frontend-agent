## Context

Unit-тесты (#0005) покрывают ядро, но не проверяют интеграцию: webview ↔ extension IPC, реальные команды VS Code, durability рана через перезапуск, поведение полной цепочки «команда → loop → tool → chat.jsonl → UI». Без e2e любой рефактор IPC или storage может тихо сломать TC-15/TC-16, и обнаружится это только при ручной проверке.

Идея — два независимых набора e2e:

- **Full (mocked)** — гоняется на каждый PR/коммит. Использует **fake OpenRouter**, который отдаёт скриптованные ответы (включая `tool_calls`). Детерминированный, быстрый, без сети, без денег. Покрывает все TC и регрессии каркаса.
- **Smoke (real models)** — гоняется руками или по расписанию (раз в день / перед релизом). Использует реальный OpenRouter ключ и реальные модели. Покрывает только 1–3 критичных сценария: «модель действительно вызывает нужные тулы». Это единственный способ поймать деградации модели/промптов.

Зачем разделение: реальные модели нестабильны (gemini-flash-lite уже подводил с `ask_user`), стоят денег и требуют сети. Если все e2e будут зависеть от модели, они быстро станут «красными по умолчанию» и их перестанут смотреть.

## Acceptance criteria

### Общая инфраструктура

- `@vscode/test-electron` + Mocha (стандартный стек для VS Code extension tests). Скрипты `test:e2e:full` и `test:e2e:smoke`.
- E2E живут в `tests/e2e/` (отдельно от unit, у них другой раннер). Каждый тест-кейс мапится на `TC-N` из `tests/manual-test-cases.md` — имя файла или describe-блок содержит `TC-N`.
- Перед каждым тестом — чистая временная workspace (через `fs.mkdtemp`), чтобы `.agents/runs/` не пересекались между тестами.

### Full suite (mocked)

- Реализован **fake OpenRouter transport**: вместо реального `fetch` подменяем функцию-клиент сценарным движком. Сценарий = массив ответов модели; на N-й запрос отдаём N-й ответ. Поддержка ответов с `tool_calls` обязательна.
- Подмена — через инъекцию (env-переменная `AI_FRONTEND_AGENT_FAKE_OPENROUTER=1` + чтение сценария из файла, или через test-only экспорт). Не править production-код хаками — единая точка подмены в `shared/openrouter/client.ts`.
- Покрыты как минимум: TC-11 (успешный цикл с tool-call), TC-12 (sandbox bypass), TC-13 (невалидные входы тула), TC-14 (лимит итераций — сценарий с 21 tool_call'ом), TC-15 (ask_user end-to-end через webview IPC), TC-16 (durability: запустили цикл → дошли до `ask_user` → симулировали «перезапуск» (выгрузка extension host'а) → дали ответ → цикл продолжился).
- Webview-проверки — через VS Code API получения panel + симулировать `postMessage`, либо через Playwright + `vscode-test-web` (решить при реализации, начать с первого варианта).
- Зелёный full-набор — обязательное условие merge'а в main (когда настроим CI).

### Smoke suite (real models)

- Использует реальный OpenRouter ключ из `process.env.OPENROUTER_API_KEY`. Если ключа нет — тест **скипается**, не падает.
- 1–3 критичных сценария:
  - «модель вызывает kb.write» — простой prompt, ожидаем хотя бы один tool_call kb.write.
  - «модель вызывает ask_user, когда не хватает данных» — prompt из smoke-команды, ожидаем хотя бы один tool_call ask_user.
- Допускается лимит на flaky: тест падает, только если из 2 попыток обе провалились (модели вероятностные).
- Прогон: руками (`npm run test:e2e:smoke`) или по расписанию (cron в CI), **не** на каждый коммит — дорого и шумно.

### Документация

- В `AGENT.md` раздел «E2E-тесты»: чем full отличается от smoke, когда что гонять, как добавить новый сценарий, как обновлять fake-сценарии при изменении промптов.
- Каждый новый TC при появлении сразу должен мапиться на full e2e (если детерминируемо) или smoke (если только реальная модель).

## Implementation notes

- Fake OpenRouter — **не библиотека nock/msw на уровне сети**, а подмена транспорта на уровне клиента. Сеть в e2e не нужна, mock на уровне HTTP — лишний слой.
- Для durability (TC-16) симулируем «перезапуск» так: после получения `ask_user` останавливаем текущий agent-loop (rejection из теста), очищаем in-memory `pendingAsks`, потом шлём `runs.userAnswer` через IPC — должно сработать `resumeRun` через registry.
- Webview-тесты тяжёлые. Если `@vscode/test-electron` + getWebview окажется муторным — допустимо первую итерацию делать «headless»: проверять только extension-сторону (storage, статусы, broadcast events), а webview-render тестировать отдельным React-unit'ом через `@testing-library/react` (это уже unit, скорее под #0005). Решить по ходу.
- Smoke-сценарии должны иметь жёсткие промпты с прямым указанием «вызови kb.write» — иначе любые улучшения общего system prompt'а делают smoke flaky.
