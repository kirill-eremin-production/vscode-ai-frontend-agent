## Context

US-12 даёт пользователю видимость по двум осям: сколько уже потратили (USD) и сколько ещё помещается в контекст конкретной модели. Без этого долгий ран — чёрный ящик, а длинные итеративные диалоги (US-10 после #0007) очень быстро упираются в лимит контекста без всякой возможности что-то с этим сделать.

Компактификация — единственная разумная защита от потери всего хода работы при подходе к лимиту: либо сжимаемся, либо ран упрётся и придётся стартовать с нуля.

## Acceptance criteria

- Парсим `usage` из ответов OpenRouter (`input_tokens`, `output_tokens`, `cached_tokens` если есть) и считаем стоимость по тарифу конкретной модели. Тарифы — литералами рядом с константами модели в `src/extension/entities/run/roles/` (по той же логике, что AGENT.md фиксирует имена моделей литералами).
- Стоимость, токены и оценка заполненности контекста сохраняются на диск (`usage.jsonl` рядом с `tools.jsonl` либо агрегаты в `meta.json` — финализируется при реализации). Накапливаются за весь жизненный цикл рана, переживают перезапуск VS Code.
- В [src/webview/features/run-list/ui/RunDetails.tsx](src/webview/features/run-list/ui/RunDetails.tsx):
  - суммарная стоимость рана + разбивка по ролям (`product`, в будущем — `architect`, `programmer`, …);
  - каждое assistant-сообщение в ленте подписано стоимостью этого шага (USD) и токенами (in/out);
  - индикатор «токенов в контексте / лимит модели» с цветовой шкалой (зелёная/жёлтая/красная зоны) — чтобы исчерпание контекста не наступало внезапно.
- Кнопка «Сжать контекст» в карточке рана (доступна в `running`/`awaiting_human`/`awaiting_user_input`):
  - перед запуском — подтверждение пользователя (необратимо для текущего `chat.jsonl`);
  - вызывает модель с инструкцией свести `chat.jsonl` + `tools.jsonl` в короткое summary;
  - текущий `chat.jsonl` бэкапится в `chat.jsonl.<timestamp>.bak` рядом, новый `chat.jsonl` начинается с записи «контекст сжат» + summary;
  - дальше ран продолжается с этого summary вместо полной истории.
- Если для модели тариф не зафиксирован — стоимость показывается как `—` с tooltip «тариф не задан», а не `0` (чтобы не вводить пользователя в заблуждение).
- Если OpenRouter не вернул `usage` — пишем 0 + warning в `tools.jsonl`, ран не ломаем.

## Implementation notes

- Тарифы держим литералами рядом с моделью: `product.ts` экспортирует `PRODUCT_MODEL`, `PRODUCT_PRICING = { inputPerMTok, outputPerMTok, cachedInputPerMTok? }`, `PRODUCT_CONTEXT_LIMIT_TOKENS`. По мере появления других ролей — тот же шаблон в их файлах.
- Учёт usage добавляется в `runAgentLoop` (#0001) — единая точка, в обход не получится. Любой шаг loop'а, после которого пришёл assistant-ответ → запись в `usage.jsonl`/`meta.json`.
- Структуру файлов под `.agents/runs/<id>/` правит только [storage.ts](src/extension/entities/run/storage.ts) (AGENT.md). Новый `usage.jsonl` или поля в `meta.json` — туда же.
- Компактификация — отдельная утилита `compactRunContext(runId)`. Какую модель использовать для сжатия (ту же, что у роли, или дешёвую) — финализируется при реализации; склоняюсь к дешёвой gemini.
- Бэкап `chat.jsonl.<timestamp>.bak` нетронут — никто его не читает в рантайме, но для аудита и git-истории пусть лежит.
- TC писать **до реализации**: TC-25 (накопление стоимости и токенов выживает перезапуск VS Code), TC-26 (ручная компактификация уменьшает токены в следующем шаге, бэкап `.bak` создан), TC-27 (модель без тарифа показывает `—`, ран не падает).
