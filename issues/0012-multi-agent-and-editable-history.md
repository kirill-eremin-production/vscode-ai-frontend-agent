## Context

После #0008 ран — это набор сессий (chat threads) с одним общим продуктом работы (brief, kb-файлы — после #0011). Сейчас сессии создаются только compact'ом, и старые сессии read-only.

Дальше нам нужно:

1. **Мульти-агент**: продукт → архитектор → программист. Между агентами идут свои чаты (`agent:product ↔ agent:architect`). Пользователь может в любой момент вмешаться в любой такой чат.
2. **Редактируемая история**: read-only в старых сессиях — это временное упрощение. По уму любой пользовательский ввод и любое сообщение в любой сессии должно быть редактируемым, со снапшотом «до» и возможностью отката. Это нужно, потому что разговор с агентом — итеративный процесс уточнений, и невозможность вернуться/поправить превращает мелкую ошибку в полное переписывание.

## Acceptance criteria

### Мульти-агент

- Сессия `kind: 'agent-agent'` помимо существующего `'user-agent'` (поле уже добавлено в #0008).
- `participants: [{kind:'agent', role:'product'}, {kind:'agent', role:'architect'}]` — структура из #0008 расширяется новой комбинацией.
- Когда роль решает «передать работу следующей роли» — extension создаёт новую `agent-agent` сессию, сидит её первым сообщением (например, brief как `from: 'agent:product'`) и запускает agent-loop архитектора.
- В UI вкладок (введён в #0008) такие сессии видны вперемешку с пользовательскими; иконка отличает по `kind`.
- Пользователь может в любой момент написать в `agent-agent` сессию — это превращает её в hybrid (`participants` дополняется `{kind:'user'}`), оба агента продолжают видеть ввод как часть истории.

### Редактируемая история

- В UI у каждого сообщения (любого `from`) есть «Edit». Редактирование пишет новое сообщение со ссылкой на предыдущее (`replaces: messageId`), оригинал остаётся в `chat.jsonl` под флагом `replacedBy: newId`.
- В ленте по умолчанию показывается актуальная версия; можно развернуть «история правок».
- Откат: кнопка «вернуть как было» восстанавливает оригинал (тоже через новую запись, не уничтожая историю).
- При резюме/новой итерации agent-loop'а в model context подаётся **актуальная** версия (что после правок).
- Снапшоты диска версионирует git, который и так трекает `.agents/`.

## Implementation notes

- Storage: `chat.jsonl` уже append-only; новая запись `{type: 'edit', id, replaces, text, at}` укладывается без миграций. Read-функция собирает «эффективное» состояние, проходя по replaces-цепочке.
- Agent-agent loops: переиспользуют `runAgentLoop`, но `userMessage` — это сообщение от соседнего агента. Никакого нового тулинга не нужно, разница только в том, что финальный assistant пишется не в `chat.jsonl` рана, а в `chat.jsonl` сессии-моста.
- Дерево сессий: `runs/<id>/meta.json` сейчас (после #0008) хранит `activeSessionId: string`. Здесь превращается в `sessions: SessionNode[]` с parent/children — для отрисовки tree, а не плоского списка вкладок.
- TC: TC-XX мульти-агент happy path (продакт → архитектор), TC-XX правка пользовательского сообщения откатывает последний шаг loop'а, TC-XX user-intervention в agent-agent сессии корректно влияет на следующий шаг.
