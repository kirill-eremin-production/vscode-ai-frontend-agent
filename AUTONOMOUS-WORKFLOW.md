# Autonomous workflow

Протокол, по которому ИИ-агент в headless-режиме выполняет задачи из `issues/` без вмешательства пользователя. Каждый запуск Claude Code получает один task id и одну фазу (`implement` или `review`). Контекст между запусками **не переживает** — состояние передаётся через git и файлы.

## Базовые правила (для обеих фаз)

- **Ветка:** работаем в `main`. Не создаём, не переключаем, не пушим.
- **Коммиты:** обязательно после каждой фазы. Если делать нечего — это всё равно ошибка, разбирайся (см. «Что если делать нечего»).
- **Никаких вопросов пользователю:** все решения принимаешь сам по AC задачи. Двусмысленность → выбери вариант, явно зафиксируй выбор в комментарии или Outcome.
- **Не пушить, не открывать PR.** Только локальные коммиты.
- **Не трогать другие задачи.** Файлы вне scope текущей задачи менять только если это прямая зависимость (например, типы общего модуля).
- **Хуки:** не используй `--no-verify`. Если pre-commit падает — чини причину.
- **Node:** всегда префиксуй npm-команды `nvm use &&` (см. `AGENT.md`).
- **Комментарии в коде:** по правилам из `AGENT.md` — на русском, JSDoc на объявлениях, объясняй «почему».
- **`.agents/`** — игнорируем на чтение/запись для целей задач (это runtime-storage пользователя).

## Что обязательно в каждой реализации

Каждая задача `0033..0052` должна породить:

1. **Код** — реализация AC.
2. **Unit-тесты** — `vitest`. Запускаются командой `nvm use && npm run test:unit`. **Должны быть зелёными перед коммитом.**
3. **User story** — добавить/обновить запись в `tests/user-stories.md` (формат «US-N. Заголовок» с секцией Acceptance). Если задача чисто инфраструктурная и пользовательской истории не порождает — явно зафиксировать это в Outcome задачи.
4. **E2E test case** — добавить `.md`-файл (ручной TC) или `.spec.ts` (автоматизированный) в `tests/e2e/specs/`. **Запускать e2e не нужно** — их прогоняет человек. Но кейс должен быть написан конкретно (steps + expected), не «проверить, что работает».
5. **Lint + build** — `nvm use && npm run lint && npm run build` зелёные перед коммитом.

Порядок проверок перед коммитом:

```
nvm use && npm run lint && npm run build && npm run test:unit
```

Если падает — чини, не коммить.

## Фаза 1 — `implement`

Один запуск Claude Code с задачей #NNNN.

1. Прочитать `issues/NNNN-*.md` целиком. Если в шапке ссылка «декомпозиция → #MMMM» — игнорируй (мы и так на подзадаче).
2. Прочитать `AGENT.md` (правила кода) и этот файл (если контекст нуждается).
3. Реализовать AC.
4. Написать unit-тесты, US, e2e TC по правилам выше.
5. Прогнать `lint + build + test:unit`. Зелёные → продолжай. Красные → чини.
6. `git add` нужных файлов (никаких `git add -A`/`git add .`).
7. `git commit` с сообщением:

   ```
   #NNNN impl: <одна строка по сути изменений>

   <2-5 строк: что сделано, какие компромиссы, ссылки на затронутые файлы>
   ```

8. Выйти. **Не** переименовывать файл в `DONE-` на этой фазе.

## Фаза 2 — `review`

Свежий запуск Claude Code, тот же task id.

1. Прочитать `issues/NNNN-*.md` целиком ещё раз.
2. `git show HEAD` — посмотреть, что было закоммичено в фазе 1.
3. Сверить с AC построчно. Что упустили, что криво, что не покрыто тестами — выписать у себя (можно прямо в чат-output).
4. Доработать. В частности обязательно проверить:
   - Каждый AC-пункт проверяется хотя бы одним unit-тестом или TC.
   - US в `tests/user-stories.md` обновлён (или явно зафиксировано отсутствие).
   - В `tests/e2e/specs/` есть TC.
   - Lint, build, test:unit зелёные.
   - Нет TODO/`// fixme`/мусора.
   - Комментарии соответствуют `AGENT.md` (русский, по делу).
5. Если потребовались правки — `git add` + `git commit`:
   ```
   #NNNN review: <что доработали>
   ```
   Если ревью показало, что всё ок и править нечего — всё равно делаем коммит-маркер пустого ревью (см. ниже), чтобы скрипт двигался дальше.
6. **Закрыть задачу:**
   - В `issues/NNNN-*.md` поменять frontmatter `status: open` → `status: done`.
   - Добавить секцию `## Outcome` (1-3 строки: что реализовано, ссылки на ключевые файлы/коммиты).
   - `git mv issues/NNNN-*.md issues/DONE-NNNN-*.md`.
7. Закоммитить переименование/закрытие отдельным коммитом (можно объединить с review-коммитом из шага 5, если он тоже был):
   ```
   #NNNN done: <короткий итог>
   ```
8. Выйти.

### Если ревью не нашло, что править

Скрипт всё равно ожидает коммит во второй фазе. В этом случае комбинируй ревью с закрытием:

- Сразу делай `status: done` + `Outcome` + `git mv` + один коммит `#NNNN done: реализация прошла ревью без замечаний`.

Так у скрипта всегда минимум 2 коммита на задачу.

## Что если делать нечего / задача уже сделана

- Если в фазе 1 видишь, что задача уже реализована (например, файл уже `DONE-`) — выйди без коммита, пусть скрипт детектирует и пропустит.
- Если в фазе 2 видишь, что фаза 1 ничего не закоммитила — это ошибка, не маскируй: попробуй сам реализовать с нуля, если успеваешь, иначе выйди без коммита.

## Ошибки и блокеры

- Если упал `npm run build` или `test:unit` и причина — баг **в твоей правке** — чини и коммить.
- Если упал из-за чужого/предыдущего кода в main — зафиксируй в Outcome, минимально объехай (например, отключи конкретный тест с TODO-комментом и ссылкой на задачу), всё равно коммить.
- **Никогда не используй `--no-verify`** или другие способы обхода хуков.
