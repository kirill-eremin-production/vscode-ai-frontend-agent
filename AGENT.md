# Agent notes

- Node.js версия зафиксирована в [.nvmrc](.nvmrc). Префиксуй npm-команды `nvm use &&`, чтобы не ловить `EBADENGINE`.
- Если работаешь в автономном прогоне (`scripts/run-tasks.sh`) — обязательно прочитай [AUTONOMOUS-WORKFLOW.md](AUTONOMOUS-WORKFLOW.md) и следуй ему буквально (фазы implement/review, обязательные unit + US + e2e TC, никаких вопросов пользователю, коммит после каждой фазы, переименование в `DONE-` только в фазе review).

## Комментарии в коде

Код пишет ИИ-агент — без подробных комментариев человек не сможет ревьюить. Это **переопределяет** дефолтное «не пишем комментарии».

- **Язык** — русский. Устоявшиеся термины (webview, publisher, lint-staged) не переводим.
- **Покрывай** функции/методы/компоненты, нетривиальные блоки, поля значимых типов, конфиги (через комментарии в соседних файлах, если формат не поддерживает inline). Формат — JSDoc над объявлениями, `//` внутри.
- **Объясняй «почему»**, а не только «что»: альтернативы, инварианты, граничные случаи. Не дублируй имя функции словами.
- **Обновляй** при правке кода — устаревший комментарий хуже отсутствующего.

## Чистота и декомпозиция

- **Размер.** ~300 строк — не растим, режем; 500+ — явный сигнал. Один файл = одна осмысленная единица.
- **Имена.** Никаких 1-2-буквенных переменных (`e`, `ev`, `ctx`). Исключения: `i`/`j` в счётчиках, `_` для unused. Везде остальное — `event`, `context`, `message`.
- **Декомпозиция функций.** Длинная функция со смысловыми этапами → именованные хелперы (имя — «что», тело — «как»). Но не плодим файлы на 5 строк ради чистоты.

## Архитектура

### Хранилище ранов: `.agents/`

Артефакты живут в `.agents/runs/<runId>/` **в корне workspace пользователя** — намеренно, чтобы коммитились вместе с кодом. globalStorage/БД отброшены: отрывают контекст от проекта.

- `meta.json` — атомарная перезапись через temp+rename. `chat.jsonl` — append-only.
- Структуру правит только [src/extension/entities/run/storage.ts](src/extension/entities/run/storage.ts). Новый файл артефакта (`plan.md`, `decisions.md`) — туда же, не размазываем fs по сервисам.
- Без открытого workspace ран не создать — `RunStorageError`, не fallback.

### Layout extension'а

ESLint-границы трактуют `src/extension/` как единый элемент, но раскладку держим FSD:

- `shared/` — инфраструктура без знаний о домене.
- `entities/` — предметная модель.
- `features/` — пользовательские сценарии.
- `providers/` — webview-провайдеры VS Code; должны быть тонкими, бизнес-логику сюда не кладём.

### IPC webview ↔ extension

Источник правды — дискриминированные union'ы в [src/extension/features/run-management/messages.ts](src/extension/features/run-management/messages.ts). Дублируются в `src/webview/shared/runs/` (ESLint запрещает кросс-импорт extension↔webview, чтобы Node-код не утёк в браузерный бандл). Правишь контракт — синхронно правишь обе стороны; разъезд поймает TS на билде.

Модель событийная (без request-id/Promise): один запрос → может породить несколько событий. `requestId` добавим, когда понадобится корреляция, не раньше.

### Секреты и модели

- Ключи — через `context.secrets` с префиксом `aiFrontendAgent.` (защита от коллизий в общем keystore).
- Имена моделей **захардкожены литералами** в местах использования. В settings не выносим, пока не нужно переключать без пересборки. Смена модели = правка константы + комментарий рядом.
- На текущем этапе только OpenRouter, ключ один.

## Стили webview'а: токены и Tailwind

С #0015 webview оформляется через семантические токены поверх vscode-переменных + Tailwind 4 (CSS-first).

- **Источник токенов** — [src/webview/app/app.css](src/webview/app/app.css). Цвета — только через `var(--vscode-*)`, иначе ломается переключение light/dark/high-contrast. Новый цвет добавляешь там же: семантическое имя (`bg-surface`, не `bg-zinc-900`) → маппинг на ближайшую vscode-переменную (с fallback'ом через запятую внутри `var()`).
- **Сборка CSS** — `npm run build:webview-css` (через `@tailwindcss/cli`, выход — `out/webview/app.css`). Подключается раньше `media/main.css`, поэтому unlayered ad-hoc-стили из `media/main.css` имеют приоритет над layered Tailwind'ом. Это даёт плавную миграцию: новые компоненты идут на токены, старые работают как есть, переписываем итеративно (#0016+).
- **В компонентах** используем utility-классы Tailwind (`inline-flex`, `gap-1`, `border`) и токенные классы (`bg-surface`, `text-foreground`, `border-border-focus`). Inline-стили и точечные цвета — нет: они ломают темы.
- **Иконки** — `lucide-react` (а не emoji). Размер задаём через `size={N}`, цвет наследуется (`currentColor`) — попадает под токен `text-*`.
- **Условная сборка классов** — `clsx` (уже в зависимостях). `tailwind-merge` сейчас не нужен: конфликтов классов нет.
- **CSS modules** — допустимы для «нестандартных» компонентов со сложной версткой/состояниями (`&:hover .child`, анимации, сетки), где Tailwind становится нечитаемым. esbuild ≥0.24 поддерживает `*.module.css` нативно (loader `local-css`), Vite не нужен. Пока инфраструктура не подключена — первый компонент, которому это понадобится, поднимает её сам: добавляет `<link>` на `out/webview/main.css` в [html.ts](src/extension/webview/html.ts) (esbuild эмитит CSS-бандл рядом с `main.js`, как только появится первый импорт `.module.css`) и `declare module '*.module.css'` в `src/webview/global.d.ts` для типов. Базовые токены и spacing внутри модулей — через `var(--color-*)` или `@apply`, чтобы темы продолжали переключаться.

## Тесты

**Unit** — `src/**/*.test.ts` рядом с кодом. Запуск: `npm run test:unit` (vitest, Node, без JSDOM). Конфиг — [vitest.config.ts](vitest.config.ts), мок `vscode` — [tests/setup-vscode.ts](tests/setup-vscode.ts). Файловые тесты гоняют реальный `os.tmpdir()`, не моки fs.

**E2E** — [tests/e2e/](tests/e2e/), Playwright + `_electron.launch()` (настоящий VS Code, не headless).

- `npm run test:e2e:full` / `:full:ui` / `:report` — прогон / интерактивный UI mode / открыть HTML-отчёт.
- **Перехват OpenRouter** живёт в отдельном [tests/e2e/test-extension/](tests/e2e/test-extension/) (второй `--extensionDevelopmentPath`). Сценарий — JSON-файл, путь через env `AI_FRONTEND_AGENT_FAKE_OPENROUTER_SCENARIO`. Прод-код о тестах не знает.
- **Изоляция:** уникальные `--user-data-dir`/`--extensions-dir`/workspace на каждый тест.
- **Видео + trace** включены всегда, аттачатся к отчёту — открой тест в HTML, увидишь .webm и DOM time-travel.
- **Версия VS Code** зафиксирована в [tests/e2e/global-setup.ts](tests/e2e/global-setup.ts) — обновляем осознанно (DOM-локаторы могут съехать).
- **Комментарии в спеках обязательны.** Под каждым DSL-вызовом и ассертом — комментарий «что и зачем», шаги нумеруем (1, 2, 5a). Спеки читают тестировщики без TS-фона. Эталон — [tc-11-tool-loop-success.spec.ts](tests/e2e/specs/tc-11-tool-loop-success.spec.ts).
- **Smoke на реальных моделях не делаем** — see #0006: e2e проверяет наше поведение при предположении контракта OpenRouter, тестирование самих моделей сюда не относится.

При правке покрытого тестом модуля — синхронно правь и тест.

## Тест-кейсы и user-stories

- [tests/user-stories.md](tests/user-stories.md) — истории US-N.
- [tests/e2e/specs/](tests/e2e/specs/) — тест-кейсы TC-N, по файлу на кейс: `.spec.ts` если автоматизирован (описание в комментарии-шапке), `.md` если пока ручной.
- [tests/checklist.md](tests/checklist.md) — smoke-чек перед коммитом.

**Правила:**

- Новая фича/роль/IPC-сообщение начинается с записи в `user-stories.md` + нового файла в `e2e/specs/`. Сначала контракт — потом код.
- Любое изменение функциональности → синхронная правка кейса и checklist'а. Устаревший кейс хуже отсутствующего.
- Номера (`US-7`, `TC-11`) сквозные, удалённые не переиспользуем. Имя файла кейса: `tc-N-<slug>`.
- Перед коммитом с правкой рантайма пройди checklist; если пункт не воспроизводится — чини код или обновляй чек-лист (с пояснением в commit message).

## Commit messages

Одно предложение на английском, начинается с `#NNNN` — id связанного тикета из `issues/`. Пример: `#0001 add tool runtime infrastructure: ...`. Жёсткая перекрёстная ссылка коммит↔тикет видна в `git log` без открытия файлов. Коммит делает пользователь, агент только формулирует.

## Трекер задач: `issues/`

Один файл = одна задача, формат — в [issues/README.md](issues/README.md).

**Заводим issue когда:** идея/требование всплыло, но не делаем сейчас; задача большая и режется на шаги (зависимости — через `## Related`); нашли баг и фиксим не сразу.

**Правила:**

- Нумерация сквозная, удалённые номера не переиспользуем.
- Закрытые issue не удаляем: статус `done`/`dropped` + `## Outcome` с итогом и ссылками на коммиты.
- Acceptance criteria мапятся на US/TC в `tests/`. Новые US/TC пишутся синхронно с реализацией.
- Перед началом работы по агентской системе сначала смотри `issues/` со `status: open` — там обдуманный задел.
