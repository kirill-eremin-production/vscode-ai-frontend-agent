# Agent notes

## Комментарии в коде

Весь код в этом репозитории пишет ИИ-агент. Чтобы человек мог быстро разобраться в происходящем и ревьюить изменения, **каждый написанный или изменённый фрагмент кода должен быть подробно прокомментирован на русском языке**. Это переопределяет дефолтное правило «по умолчанию не писать комментарии».

Правила:

- **Язык.** Все комментарии — на русском. Исключение: устоявшиеся английские термины (webview, publisher, lint-staged и т.п.) не переводим.
- **Подробность.** Комментируй не только «что делает код», но и **почему** именно так: какие альтернативы были отброшены, какие инварианты соблюдаются, где граничные случаи.
- **Покрытие.** Комментарий должен быть у:
  - каждой функции / метода / React-компонента (назначение, входы, выходы, побочные эффекты);
  - нетривиальных блоков внутри функции (условия, циклы, async-операции, обработка ошибок);
  - каждого поля в значимых интерфейсах / типах;
  - конфигурационных секций (eslint, tsconfig, package.json scripts и т.п.) — через комментарии в соседних файлах, если формат сам по себе не поддерживает комментарии (например, рядом с `package.json` — в README или AGENT.md).
- **Формат.** Для TypeScript/JavaScript используй JSDoc (`/** ... */`) над объявлениями и обычные `//` для пояснений внутри функций. Для JSON-файлов, которые не поддерживают комментарии, описывай назначение полей в сопроводительной документации.
- **Актуальность.** При изменении кода обязательно обновляй и комментарии — устаревший комментарий хуже отсутствующего.
- **Без воды.** Не дублируй само имя функции словами («функция `getUser` получает пользователя»). Объясняй контекст, который не виден из сигнатуры.

## Архитектура мульти-агентной системы

### Хранилище ранов: `.agents/`

Артефакты ранов лежат в `.agents/runs/<runId>/` **в корне открытого workspace пользователя** — намеренно, чтобы документация и решения агентов жили рядом с кодом и могли коммититься в git вместе с ним. Альтернативы (globalStorage, БД) отброшены: они отрывают контекст от проекта.

- `meta.json` — атомарно перезаписывается через temp-файл + rename.
- `chat.jsonl` — append-only, по строке на сообщение.
- Структуру правит только `src/extension/entities/run/storage.ts`. Если нужно добавить новый файл артефакта (`plan.md`, `decisions.md`) — добавляй туда, не размазывай fs-логику по сервисам.
- Ран без открытого workspace создать нельзя — это явная ошибка `RunStorageError`, не fallback.

### Layout extension-стороны

ESLint-границы (`eslint.config.mjs`) трактуют `src/extension/` как **единый элемент** — внутренние импорты не проверяются. Тем не менее держим FSD-подобную раскладку по соглашению:

- `shared/` — инфраструктура без знаний о домене (OpenRouter client, secrets, nonce).
- `entities/` — предметная модель (`run/`: types, storage, service, title).
- `features/` — пользовательские сценарии (`run-management/`: IPC-контракт + wire).
- `providers/` — webview-провайдеры VS Code (sidebar, panel).

При добавлении новой роли/инструмента сначала спроси себя, в какой слой это попадает. Не клади бизнес-логику в `providers/` — они должны быть тонкими.

### IPC webview ↔ extension

Контракт — дискриминированные union-ы в `src/extension/features/run-management/messages.ts`. Это **источник правды**.

В webview типы и команды дублируются в `src/webview/shared/runs/` (ESLint-границы запрещают импорт из extension в webview и наоборот — это страхует от утечки Node-кода в браузерный бандл). При изменении контракта правь обе стороны одновременно; разъезд поймает TypeScript на ближайшем билде.

Модель сообщений событийная (никаких request-id/Promise): один запрос может породить несколько событий (`runs.created` + обновлённый `runs.list.result`). Если когда-нибудь понадобится корреляция запрос/ответ — добавим `requestId`, но не раньше реальной нужды.

### Секреты и ключи моделей

- Все ключи — через `context.secrets` с префиксом `aiFrontendAgent.` (защита от коллизий с другими расширениями в общем keystore).
- Имена моделей **захардкожены литералами** в соответствующих модулях (`title.ts` и т.д.). Не выносим в settings, пока не появится явная необходимость их переключать без пересборки. При смене модели — правь константу и комментарий рядом.
- На текущем этапе используется только OpenRouter; ключ один.
